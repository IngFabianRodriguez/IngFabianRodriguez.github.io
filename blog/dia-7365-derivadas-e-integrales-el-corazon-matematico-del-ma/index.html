Día 7: Derivadas e Integrales - El Corazón Matemático del Machine Learning

🎯 Por Qué Este Tema No Es "Matemáticas de Relleno"

Parcero, si hay algo que separa a un ingeniero que "usa librerías" de uno que entiende qué carajos está pasando bajo el capó, son las derivadas e integrales.

En ML, todo—y digo todo—se reduce a optimización. Y optimizar sin entender derivadas es como manejar un carro sin saber para qué sirve el freno. Tal vez llegues, pero vas a chocar feo.

Las derivadas te dicen hacia dónde moverte para minimizar el error. Las integrales te permiten acumular información (piensa en probabilidades, áreas bajo curvas ROC, expectativas). Sin estas herramientas, estás ciego.

📐 Fundamento Matemático: ¿Qué Es Una Derivada?

La derivada mide la tasa de cambio instantánea de una función. En términos de ML, es la respuesta a: "Si muevo este peso de mi red neuronal un poquito, ¿cuánto cambia mi error?"

Definición Formal

$$f'(x) = \lim_{h \to 0} \frac{f(x+h) - f(x)}{h}$$

Esto te dice: "Qué tanto sube o baja $f$ cuando $x$ se mueve una cantidad infinitesimal $h$".

Ejemplo Concreto: Función de Pérdida

Imagina que tu loss function es:

$$\mathcal{L}(w) = (y - wx)^2$$

Donde $w$ es el peso que queremos ajustar. La derivada con respecto a $w$ es:

$$\frac{\partial \mathcal{L}}{\partial w} = \frac{\partial}{\partial w}(y - wx)^2 = 2(y - wx) \cdot (-x) = -2x(y - wx)$$

¿Qué significa esto?

Este gradiente te dice en qué dirección y qué tan rápido debes ajustar $w$ para reducir el error. Si ∂w/∂L>0 , disminuye $w$. Si es negativo, auméntalo.

🔧 Analogía de Trinchera: El Barista Optimizando el Tinto

Imagina que eres barista y quieres encontrar la temperatura perfecta para el tinto que le sirves al equipo.

Muy frío = quejas. Muy caliente = quemados.

  • La función de pérdida es el nivel de quejas del equipo.
  • La derivada es cómo cambian las quejas cuando subes o bajas 1°C.
  • Gradient descent es ir ajustando la temperatura poco a poco hasta que las quejas sean mínimas.

Si la derivada es negativa (subir temperatura reduce quejas), le subes. Si es positiva, le bajas. Así optimizas.

En ML, estás ajustando millones de "temperaturas" (pesos) al mismo tiempo. Por eso necesitas calcular derivadas eficientemente.

🧮 Reglas de Derivación Esenciales

1. Regla de la Potencia

$$\frac{d}{dx}(x^n) = nx^{n-1}$$

2. Regla de la Cadena (Chain Rule) - LA MÁS CRÍTICA EN ML

Si $y = f(g(x))$, entonces:

$$\frac{dy}{dx} = \frac{df}{dg} \cdot \frac{dg}{dx}$$

¿Por qué es crítica? Porque backpropagation en redes neuronales ES aplicar la regla de la cadena repetidamente a través de capas.

📊 Integrales: Acumulando Información

La integral es la operación inversa de la derivada. Si la derivada mide cambio, la integral mide acumulación.

Definición

$$\int_a^b f(x) \, dx = \lim_{n \to \infty} \sum_{i=1}^n f(x_i) \Delta x$$

Esto es el área bajo la curva de $f(x)$ entre $a$ y $b$

Aplicación en ML: Esperanza Matemática

La esperanza de una variable aleatoria continua $X$ con densidad $p(x)$ es:

$$\mathbb{E}[X] = \int_{-\infty}^{\infty} x \cdot p(x) \, dx$$

Esto aparece en VAEs (Variational Autoencoders), Bayesian ML, y cualquier modelo probabilístico.

class CalculusError(Exception):
    """Excepción base para errores de cálculo."""
    pass

# ==================== DATACLASSES ====================

@dataclass(frozen=True, slots=True)  # Optimización de memoria
class DerivativeResult:
    """
    Resultado de una operación de derivación.
    
    ¿Por qué frozen=True?
    - Inmutabilidad garantiza thread-safety
    - Permite usar como key en diccionarios
    
    ¿Por qué slots=True?
    - Reduce overhead de memoria ~40% vs dict-based attrs
    - Crítico cuando calculas 1M+ derivadas en batch
    """
    expression: Expr
    variable: Symbol
    derivative: Expr
    simplified: Expr
    order: int = 1
    
    def to_callable(self, *variables: Symbol) -> Callable:
        """
        Convierte la derivada simbólica en función Python evaluable.
        
        Esto es CLAVE para integración con NumPy/PyTorch:
        - SymPy calcula la derivada (exacta, simbólica)
        - lambdify la convierte en código Python optimizado
        - La usas en tu loop de entrenamiento
        """
        try:
            return lambdify(variables, self.simplified, modules='numpy')
        except Exception as e:
            logger.error(f"Error convirtiendo a callable: {e}")
            raise CalculusError(f"No se pudo convertir {self.simplified}") from e


@dataclass(slots=True)
class IntegralResult:
    """Resultado de una operación de integración."""
    expression: Expr
    variable: Symbol
    integral: Expr
    definite: bool = False
    lower_bound: float | None = None
    upper_bound: float | None = None
    
    def evaluate(self) -> float | Expr:
        """
        Evalúa integral definida numéricamente.
        
        ¿Por qué separar evaluate() del constructor?
        - La integración simbólica puede ser costosa (O(n²) en complejidad)
        - Permite lazy evaluation: solo calculas cuando necesitas el valor
        - Útil para pipelines batch donde generas 1000 integrales y solo evalúas 10
        """
        if not self.definite:
            logger.warning("Integral indefinida, retornando expresión simbólica")
            return self.integral
        
        if self.lower_bound is None or self.upper_bound is None:
            raise CalculusError("Límites de integración no definidos")
        
        try:
            # Sustituir límites y evaluar
            result = self.integral.subs(self.variable, self.upper_bound) - \
                     self.integral.subs(self.variable, self.lower_bound)
            return float(result.evalf())
        except Exception as e:
            logger.error(f"Error evaluando integral: {e}")
            raise CalculusError(f"Fallo al evaluar integral definida") from e


# ==================== MOTOR DE CÁLCULO ====================

class SymbolicCalculusEngine:
    """
    Motor de cálculo simbólico optimizado para ML/AI.
    
    Casos de uso reales:
    1. Derivar loss functions custom antes de implementarlas en PyTorch
    2. Validar matemáticamente transformaciones de features
    3. Generar código de gradientes para optimizadores personalizados
    4. Calcular integrales en modelos probabilísticos (VAE, Bayesian NN)
    
    ¿Por qué no usar directamente autograd de PyTorch/TensorFlow?
    - Autograd es numérico y aproximado
    - Esto es simbólico y exacto
    - Úsalo en fase de diseño/validación, luego migra a autograd para producción
    """
    
    __slots__ = ('_cache', '_max_cache_size')  # Previene dict overhead
    
    def __init__(self, max_cache_size: int = 1000):
        """
        Args:
            max_cache_size: Límite de expresiones cacheadas (previene memory leak)
        """
        self._cache: Dict[tuple, DerivativeResult | IntegralResult] = {}
        self._max_cache_size = max_cache_size
        
logger.info
(f"SymbolicCalculusEngine inicializado (cache_size={max_cache_size})")
    
    def _validate_expression(self, expr: Expr, variables: List[Symbol]) -> None:
        """
        Validación previa para evitar explosiones de complejidad.
        
        SymPy puede colgarse con expresiones patológicas (ej: 1000 términos nested).
        Esto detecta casos problemáticos ANTES de intentar derivar.
        """
        free_symbols = 
expr.free
_symbols
        
        # Verificar que las variables existen en la expresión
        for var in variables:
            if var not in free_symbols:
                raise VariableNotFoundError(
                    f"Variable '{var}' no encontrada en expresión. "
                    f"Variables disponibles: {free_symbols}"
                )
        
        # Detectar expresiones excesivamente complejas
        expr_str = str(expr)
        if len(expr_str) > 10000:  # Heurística: >10K chars = probablemente problemático
            logger.warning(
                f"Expresión muy compleja ({len(expr_str)} chars). "
                f"Esto puede causar lentitud."
            )
    
    def compute_derivative(
        self,
        expression: Expr,
        variable: Symbol,
        order: int = 1,
        simplify_result: bool = True
    ) -> DerivativeResult:
        """
        Calcula derivada de orden n con caching inteligente.
        
        Complejidad:
        - Mejor caso (cache hit): O(1)
        - Peor caso (miss + simplify): O(n²) donde n = términos en expresión
        
        Args:
            expression: Expresión simbólica SymPy
            variable: Variable respecto a la cual derivar
            order: Orden de derivación (1=primera, 2=segunda, etc.)
            simplify_result: Si True, simplifica algebraicamente (costoso pero legible)
        
        Returns:
            DerivativeResult con la derivada calculada
        
        Raises:
            InvalidExpressionError: Si la expresión no es derivable
            VariableNotFoundError: Si la variable no existe en la expresión
        """
        # Cache key: hash de (expresión, variable, orden)
        cache_key = (hash(expression), hash(variable), order)
        
        if cache_key in self._cache:
            logger.debug(f"Cache HIT para derivada de orden {order}")
            return self._cache[cache_key]
        
        
logger.info
(f"Calculando derivada de orden {order} respecto a {variable}")
        
        try:
            self._validate_expression(expression, [variable])
            
            # Calcular derivada iterativamente (SymPy optimiza esto internamente)
            derivative = expression
            for _ in range(order):
                derivative = diff(derivative, variable)
            
            # Simplificar si se solicita (esto puede ser MUY costoso)
            simplified = simplify(derivative) if simplify_result else derivative
            
            result = DerivativeResult(
                expression=expression,
                variable=variable,
                derivative=derivative,
                simplified=simplified,
                order=order
            )
            
            # Cachear con límite de tamaño (LRU implícito)
            if len(self._cache) >= self._max_cache_size:
                # Evicción simple: eliminar entrada más antigua
                self._cache.pop(next(iter(self._cache)))
            
            self._cache[cache_key] = result
            
logger.info
(f"Derivada calculada exitosamente (cache_size={len(self._cache)})")
            
            return result
        
        except Exception as e:
            logger.error(f"Error calculando derivada: {e}")
            raise InvalidExpressionError(f"No se pudo derivar la expresión") from e
    
    def compute_integral(
        self,
        expression: Expr,
        variable: Symbol,
        lower_bound: float | None = None,
        upper_bound: float | None = None
    ) -> IntegralResult:
        """
        Calcula integral (definida o indefinida).
        
        Complejidad:
        - Integral indefinida: O(n²) típicamente
        - Integral definida: O(n²) + O(n) para evaluación
        
        ¿Cuándo usar definida vs indefinida?
        - Indefinida: Para obtener la antiderivada simbólica
        - Definida: Para calcular áreas, probabilidades, expectativas
        
        Ejemplo ML:
            # Esperanza de distribución normal
            x = sp.Symbol('x', real=True)
            pdf = (1/sp.sqrt(2*sp.pi)) * sp.exp(-x**2 / 2)
            expectation = engine.compute_integral(x * pdf, x, -sp.oo, sp.oo)
        """
        is_definite = lower_bound is not None and upper_bound is not None
        
        
logger.info
(
            f"Calculando integral {'definida' if is_definite else 'indefinida'} "
            f"respecto a {variable}"
        )
        
        try:
            self._validate_expression(expression, [variable])
            
            # Calcular antiderivada
            antiderivative = integrate(expression, variable)
            
            result = IntegralResult(
                expression=expression,
                variable=variable,
                integral=antiderivative,
                definite=is_definite,
                lower_bound=lower_bound,
                upper_bound=upper_bound
            )
            
            
logger.info
("Integral calculada exitosamente")
            return result
        
        except Exception as e:
            logger.error(f"Error calculando integral: {e}")
            raise InvalidExpressionError(f"No se pudo integrar la expresión") from e
    
    def compute_gradient(
        self,
        expression: Expr,
        variables: List[Symbol]
    ) -> Dict[Symbol, Expr]:
        """
        Calcula gradiente (vector de derivadas parciales).
        
        Matemáticamente:
        $$\nabla f(x_1, x_2, ..., x_n) = \left[ \frac{\partial f}{\partial x_1}, 
                                               \frac{\partial f}{\partial x_2}, 
                                               ..., 
                                               \frac{\partial f}{\partial x_n} \right]$$
        
        Esto es EXACTAMENTE lo que hace backprop, pero simbólicamente.
        
        Uso típico:
            # Loss MSE: (y - (w1*x1 + w2*x2 + b))^2
            w1, w2, b, x1, x2, y = sp.symbols('w1 w2 b x1 x2 y')
            loss = (y - (w1*x1 + w2*x2 + b))**2
            
            grad = engine.compute_gradient(loss, [w1, w2, b])
            # grad = {w1: -2*x1*(y - w1*x1 - w2*x2 - b),
            #         w2: -2*x2*(y - w1*x1 - w2*x2 - b),
            #         b: -2*(y - w1*x1 - w2*x2 - b)}
        """
        
logger.info
(f"Calculando gradiente respecto a {len(variables)} variables")
        
        self._validate_expression(expression, variables)
        
        gradient = {}
        for var in variables:
            try:
                # Reutilizar compute_derivative para aprovechar cache
                deriv_result = self.compute_derivative(expression, var, order=1)
                gradient[var] = deriv_result.simplified
            except Exception as e:
                logger.error(f"Error calculando ∂f/∂{var}: {e}")
                raise CalculusError(f"Fallo al calcular gradiente") from e
        
        
logger.info
(f"Gradiente calculado con {len(gradient)} componentes")
        return gradient
    
    def compute_hessian(
        self,
        expression: Expr,
        variables: List[Symbol]
    ) -> Dict[tuple[Symbol, Symbol], Expr]:
        """
        Calcula matriz Hessiana (derivadas segundas).
        
        Matemáticamente:
        $$H_{ij} = \frac{\partial^2 f}{\partial x_i \partial x_j}$$
        
        ¿Para qué sirve?
        - Determinar convexidad de la loss (si Hessian > 0, es convexa)
        - Métodos de optimización de segundo orden (Newton, L-BFGS)
        - Análisis de sensibilidad de parámetros
        
        Complejidad: O(n²) donde n = número de variables
        """
        
logger.info
(f"Calculando Hessiana ({len(variables)}x{len(variables)})")
        
        self._validate_expression(expression, variables)
        
        hessian = {}
        for i, var_i in enumerate(variables):
            for j, var_j in enumerate(variables):
                try:
                    # Derivar dos veces: primero respecto a var_i, luego a var_j
                    first_deriv = self.compute_derivative(expression, var_i).simplified
                    second_deriv = self.compute_derivative(first_deriv, var_j).simplified
                    
                    hessian[(var_i, var_j)] = second_deriv
                except Exception as e:
                    logger.error(f"Error calculando ∂²f/∂{var_i}∂{var_j}: {e}")
                    raise CalculusError(f"Fallo al calcular Hessiana") from e
        
        
logger.info
(f"Hessiana calculada con {len(hessian)} entradas")
        return hessian

🎯 Alivio del Dolor: Por Qué Esto Te Salva en Producción

Problema #1: "Mi modelo no converge y no sé por qué"

Solución: Usa compute_gradient() para derivar simbólicamente tu loss custom. Si encuentras términos que explotan (ej: $\frac{1}{x}$ con $x to 0$), ya sabes dónde está el problema.

Problema #2: "Implementé un optimizador nuevo y da resultados raros"

Solución: Valida matemáticamente tus update rules con SymPy ANTES de codificarlas. Si la derivada simbólica dice una cosa y tu código hace otra, el bug es tuyo.

Problema #3: "Necesito calcular la curvatura de mi loss para convergencia adaptativa"

Solución: Usa compute_hessian(). Si la Hessiana tiene eigenvalues negativos, tu loss NO es convexa en esa región. Ajusta tu arquitectura o regularización.

Problema #4: "Los valores de mi VAE no tienen sentido matemático"

Solución: Calcula simbólicamente las integrales de tu prior y posterior. Si no coinciden con las aproximaciones numéricas de TensorFlow, hay un bug de implementación.

🔥 Reglas de Optimización para No Quedar Embalado

1. Simplifica Solo Si Es Necesario

# MAL: Siempre simplifica (costoso)
grad = engine.compute_derivative(expr, var, simplify_result=True)

# BIEN: Simplifica solo para debugging/logging
if __debug__:
    grad = engine.compute_derivative(expr, var, simplify_result=True)
else:
    grad = engine.compute_derivative(expr, var, simplify_result=False)

Por qué: simplify() en SymPy puede tardar segundos en expresiones grandes. En un loop de 10,000 iteraciones, eso son horas perdidas.

2. Usa Caching Agresivamente

El SymbolicCalculusEngine ya tiene cache interno, pero puedes hacer más:

# Pre-computar todas las derivadas que vas a necesitar
derivatives = {
    var: engine.compute_derivative(loss, var)
    for var in all_variables
}

# Reutilizar en loop
for epoch in range(1000):
    for var in all_variables:
        grad = derivatives[var].to_callable(...)  # O(1) lookup

📚 Complejidad Computacional: Cuándo Usarlo y Cuándo No

Operación Complejidad Cuándo Usar
diff(expr, var) $O(n)$ a $O(n^2)$ Siempre (es rápido)
simplify(expr) $O(n^2)$ a $O(2^n)$ Solo debugging
integrate(expr, var) $O(n^2)$ a $O(\infty)$ Validación offline
compute_gradient() $O(kn)$ $(k = vars)$ Fase de diseño
compute_hessian() $O(k^2 n)$ Análisis de convergencia

Regla de oro: Usa SymPy para validar, no para ejecutar. En producción, autograd (PyTorch/TensorFlow) es 1000x más rápido.
