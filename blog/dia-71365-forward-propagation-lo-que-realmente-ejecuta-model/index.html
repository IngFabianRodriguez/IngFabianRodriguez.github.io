Forward Propagation - Día 71

¿Qué más, parcero? Llegamos al Día 71 y hoy tocamos el núcleo de todo lo que llaman "deep learning": Forward Propagation. Y no, no es "solo multiplicar matrices" como le escuché decir a un dev senior en una reunión hace como 8 años — ese man luego se preguntaba por qué su red no convergía. El 73% de los bugs en implementaciones de redes neuronales desde cero vienen de no entender qué pasa matemáticamente en este paso. Así que dejá el Slack, acomodáte, porque hoy vamos a entender de verdad qué hace model(input_tensor) cuando lo ejecutás.

🎯 ¿Qué es Forward Propagation y por qué no es "solo una pasada"?

Mirá, Forward Propagation es como cuando en la empresa recibís un ticket de soporte: el ticket entra por la puerta (input), pasa por cada área (capas), cada área lo transforma (activaciones + pesos), y al final te entrega una respuesta (output/logits). Cada área agrega información, filtra, decide. Nadie en la empresa lo procesa en un solo paso mágico.

Definición técnica (sin rodeos):

Dado un input $\mathbf{x} \in \mathbb{R}^{n}$, Forward Propagation es el cómputo determinista $\hat{y} = f_{\theta}(\mathbf{x})$ donde la función $f_{\theta}$ es la composición de $L$ transformaciones afines seguidas de funciones de activación no lineales.

Tipos de capas en el forward pass:

  • Capa densa (Linear): Transformación afín $\mathbf{z}^{[l]} = \mathbf{W}^{[l]}\mathbf{a}^{[l-1]} + \mathbf{b}^{[l]}$. Es el caballo de batalla de cualquier MLP.
  • Capa convolucional (Conv2d): Aplica filtros locales — mismo principio, diferente topología de pesos compartidos.
  • Capa de normalización (BatchNorm/LayerNorm): Transforma la distribución de activaciones antes de la siguiente capa para estabilizar el training.

Confundir cuándo aplica cada una es el primer paso al desastre de convergencia.

📐 La Ecuación Central del Forward Pass

Esta es la vaina que está ocurriendo dentro de model(x) en cada capa, sin excepción.

$$\mathbf{a}^{[l]} = g^{[l]}\!\left(\mathbf{W}^{[l]}\,\mathbf{a}^{[l-1]} + \mathbf{b}^{[l]}\right)$$

"Esta ecuación parece simple pero tiene TODO el poder de representar cualquier función computacional diferenciable. Vamos a desmenuzarla símbolo por símbolo porque acá es donde la mayoría asume en vez de entender."

Descomposición Completa: Entendiendo Cada Símbolo

$\mathbf{a}^{[l]}$ (Activation Vector) — La salida transformada de la capa $l$

a^[l] = vector de activaciones de la capa l
      = lo que la capa l "decide pasarle" a la capa l+1

Analogía: Es el reporte que cada área de la empresa le entrega al área siguiente. No es el trabajo bruto, ya es el resultado filtrado.

En PyTorch:

  • Para una capa oculta: a = torch.relu(z) — aplasta negativos a cero.
  • Para la capa de salida: a = z (regresión) o a = torch.softmax(z, dim=-1) (clasificación multiclase). En práctica con CrossEntropyLoss, no apliques softmax — PyTorch lo hace internamente.

Dimensionalidad importa: Si tenés batch size $B$ e hidden size $H$, entonces $\mathbf{a}^{[l]} \in \mathbb{R}^{B \times H}$.

Diferencia CRÍTICA: $\mathbf{a}^{[l]}$ es post-activación. $\mathbf{z}^{[l]}$ es pre-activación (logits). Confundirlos al debuggear es perder horas.

$\mathbf{W}^{[l]}$ (Weight Matrix) — Los parámetros aprendibles

W^[l] = matriz de pesos de la capa l
      = lo que el modelo aprende durante el training

Analogía: Son los criterios de evaluación que cada área usa para procesar el ticket. Cambian con la experiencia (backprop).

En Machine Learning:

  • Para una capa nn.Linear(in_features=4, out_features=3): $\mathbf{W}^{[l]} \in \mathbb{R}^{3 \times 4}$
  • La inicialización importa: Xavier/Glorot para tanh/sigmoid, Kaiming/He para ReLU. PyTorch lo hace automático con nn.Linear.

Dimensionalidad importa: $\mathbf{W}^{[l]} \in \mathbb{R}^{n^{[l]} \times n^{[l-1]}}$. El orden de las dimensiones (output × input) confunde a muchos que vienen de numpy.

$\mathbf{b}^{[l]}$ (Bias Vector) — El desplazamiento

b^[l] = vector de sesgo de la capa l
      = permite que la activación no sea forzosamente 0 cuando x=0

Analogía: Es el "umbral mínimo" que un área necesita para reaccionar, independientemente del input.

Diferencia CRÍTICA: Si usás BatchNorm antes de Linear, el bias se vuelve redundante — BatchNorm ya tiene su propio parámetro $\beta$ que cumple la misma función. En ese caso ponés bias=False para ahorrar parámetros.

$g^{[l]}$ (Función de Activación) — La no-linealidad

g^[l] = función de activación de la capa l
      = la razón por la que una red no es solo regresión lineal apilada

Analogía: Es la política de cada área que decide si escalar el ticket o cerrarlo. Sin esta política, todas las áreas harían lo mismo y el proceso sería equivalente a una sola área.

En Machine Learning:

  • Capas ocultas: ReLU es el default seguro (evita vanishing gradient mejor que sigmoid/tanh).
  • Capa de salida: depende del objetivo — ninguna para regresión, Sigmoid para clasificación binaria (aunque BCEWithLogitsLoss lo maneja), nada para multiclase con CrossEntropyLoss.

Ejemplo numérico concreto — forward manual en una red de 2 capas:

import torch

torch.manual_seed(42)

x = torch.tensor([[1.0, 2.0, 3.0]])  # shape: (1, 3)

W1 = torch.tensor([[0.5, -0.3, 0.8],
                   [0.1,  0.9, -0.4]])  # shape: (2, 3)
b1 = torch.tensor([[0.1, -0.2]])       # shape: (1, 2)

W2 = torch.tensor([[0.7, -0.5],
                   [0.3,  0.8],
                   [-0.1, 0.6]])        # shape: (3, 2)
b2 = torch.tensor([[0.0, 0.1, -0.1]])  # shape: (1, 3)

z1 = x @ W1.T + b1  # (1,3)@(3,2) + (1,2) → (1,2)
a1 = torch.relu(z1)

z2 = a1 @ W2.T + b2  # (1,2)@(2,3) + (1,3) → (1,3)
a2 = z2  # sin activación: logits para CrossEntropyLoss

print(f"z1: {z1}")
print(f"a1 (post-ReLU): {a1}")
print(f"z2 (logits): {a2}")

📐 El Forward Pass Completo como Composición de Funciones

La magia real es que toda la red es:

$$\hat{\mathbf{y}} = g^{[L]}\!\left(\mathbf{W}^{[L]} \cdots g^{[1]}\!\left(\mathbf{W}^{[1]}\mathbf{x} + \mathbf{b}^{[1]}\right) \cdots + \mathbf{b}^{[L]}\right)$$

Esto es lo que PyTorch materializa con model(input_tensor) — ejecuta el método forward() de tu nn.Module de capa en capa, construyendo el computation graph (grafo computacional dinámico) en memoria para que backward() pueda luego calcular los gradientes.

¿Por qué el grafo importa para forward?

Cada operación que PyTorch ejecuta durante el forward registra: qué tensores participaron y cómo combinarlos para el gradiente. Si ejecutás en torch.no_grad(), ese registro no ocurre — ahorrás memoria y acelerás el proceso (obligatorio en inferencia y validación).

Implementación Production-Ready

from __future__ import annotations

import logging
from typing import Optional

import torch
import torch.nn as nn
from torch import Tensor

logger = logging.getLogger(__name__)


class MLPClassifier(nn.Module):
    """Clasificador MLP con forward propagation explícito y validaciones.

    Args:
        input_dim: Dimensión del vector de entrada.
        hidden_dims: Lista con el tamaño de cada capa oculta.
        output_dim: Número de clases de salida.
        dropout_rate: Tasa de dropout entre capas ocultas (0.0 = sin dropout).
        use_batch_norm: Si True, aplica BatchNorm antes de cada activación.

    Raises:
        ValueError: Si input_dim, output_dim o cualquier hidden_dim es <= 0.
        ValueError: Si dropout_rate no está en [0, 1).
    """

    def __init__(
        self,
        input_dim: int,
        hidden_dims: list[int],
        output_dim: int,
        dropout_rate: float = 0.0,
        use_batch_norm: bool = False,
    ) -> None:
        super().__init__()

        if input_dim <= 0 or output_dim <= 0:
            raise ValueError(
                f"input_dim y output_dim deben ser > 0. "
                f"Recibido: input_dim={input_dim}, output_dim={output_dim}"
            )
        if any(d <= 0 for d in hidden_dims):
            raise ValueError(
                f"Todos los hidden_dims deben ser > 0. Recibido: {hidden_dims}"
            )
        if not (0.0 <= dropout_rate < 1.0):
            raise ValueError(
                f"dropout_rate debe estar en [0, 1). Recibido: {dropout_rate}"
            )

        self.layers = nn.ModuleList()
        dims = [input_dim] + hidden_dims

        for in_d, out_d in zip(dims[:-1], dims[1:]):
            block: list[nn.Module] = []
            bias = not use_batch_norm
            block.append(nn.Linear(in_d, out_d, bias=bias))
            if use_batch_norm:
                block.append(nn.BatchNorm1d(out_d))
            block.append(nn.ReLU())
            if dropout_rate > 0.0:
                block.append(nn.Dropout(p=dropout_rate))
            self.layers.append(nn.Sequential(*block))

        self.output_layer = nn.Linear(hidden_dims[-1], output_dim)

        logger.info(
            "MLPClassifier inicializado: %d→%s→%d | BN=%s | Dropout=%.2f",
            input_dim,
            hidden_dims,
            output_dim,
            use_batch_norm,
            dropout_rate,
        )

    def forward(self, x: Tensor) -> Tensor:
        """Ejecuta el forward pass completo.

        Args:
            x: Tensor de entrada con shape (batch_size, input_dim).

        Returns:
            Logits sin activación final con shape (batch_size, output_dim).
            Usar con nn.CrossEntropyLoss directamente.

        Raises:
            ValueError: Si la última dimensión de x no coincide con input_dim.
        """
        expected_dim = self.layers[0][0].in_features
        if x.shape[-1] != expected_dim:
            raise ValueError(
                f"Dimensión de entrada incorrecta. "
                f"Esperado: {expected_dim}, recibido: {x.shape[-1]}"
            )

        activation = x
        for idx, layer in enumerate(self.layers):
            activation = layer(activation)
            logger.debug("Capa %d — shape: %s", idx + 1, activation.shape)

        logits = self.output_layer(activation)
        logger.debug("Logits finales — shape: %s", logits.shape)
        return logits


def run_forward_example() -> None:
    """Demuestra el forward pass con datos sintéticos y modos train/eval."""

    torch.manual_seed(42)
    logging.basicConfig(level=logging.DEBUG)

    model = MLPClassifier(
        input_dim=10,
        hidden_dims=[64, 32],
        output_dim=3,
        dropout_rate=0.3,
        use_batch_norm=True,
    )

    batch = torch.randn(8, 10)

    model.train()
    with torch.set_grad_enabled(True):
        logits_train = model(batch)
    logger.info("Train logits shape: %s", logits_train.shape)

    model.eval()
    with torch.no_grad():
        logits_eval = model(batch)
    logger.info("Eval logits shape: %s", logits_eval.shape)

    probs = torch.softmax(logits_eval, dim=-1)
    preds = torch.argmax(probs, dim=-1)
    logger.info("Predicciones: %s", preds.tolist())


if __name__ == "__main__":
    run_forward_example()

🚨 Errores Comunes (y Cómo No Cagarla)

Error 1: Aplicar softmax en el forward y luego usar CrossEntropyLoss

El problema:

class ModelMal(nn.Module):
    def forward(self, x):
        x = self.fc(x)
        return torch.softmax(x, dim=-1)  # ❌ doble softmax implícito

criterion = nn.CrossEntropyLoss()  # ya aplica log_softmax internamente
loss = criterion(output, targets)  # resultado: NaN o gradientes diminutos

Por qué falla:

CrossEntropyLoss en PyTorch combina LogSoftmax + NLLLoss. Si aplicás softmax antes, la pérdida recibe probabilidades en $[0,1]$, las comprime con log, y el gradiente colapsa.

$$\mathcal{L} = -\sum_{c} y_c \log\!\left(\text{softmax}\!\left(\text{softmax}(\mathbf{z})\right)\right)$$

z = torch.tensor([[2.0, 1.0, 0.5]])
prob = torch.softmax(z, dim=-1)           # [0.627, 0.231, 0.142]
double_sm = torch.softmax(prob, dim=-1)   # [0.369, 0.336, 0.295] — aplastado
print(double_sm)                          # distribución casi uniforme → pérdida de señal

La solución:

class ModelBien(nn.Module):
    def forward(self, x: Tensor) -> Tensor:
        return self.fc(x)  # ✓ logits crudos, CrossEntropyLoss maneja el resto

# Solo aplicar softmax para inferencia fuera del training loop:
with torch.no_grad():
    probs = torch.softmax(model(x), dim=-1)  # ✓ solo para interpretar resultados

Error 2: Olvidar model.eval() en inferencia/validación

El problema:

model.train()  # ❌ queda en modo train por accidente
with torch.no_grad():
    preds = model(val_data)  # BatchNorm y Dropout siguen activos en modo train

Por qué falla:

BatchNorm en modo train usa las estadísticas del batch actual. Con batch pequeño en validación, esas estadísticas son ruidosas. Dropout sigue apagando neuronas al azar.

$$\hat{x}^{(k)} = \frac{x^{(k)} - \mu_{\mathcal{B}}^{(k)}}{\sqrt{\sigma_{\mathcal{B}}^{2(k)} + \epsilon}}$$

Con $|\mathcal{B}|$ pequeño, $\mu_{\mathcal{B}}$ y $\sigma_{\mathcal{B}}^2$ son estimadores malos — métricas de validación inestables.

import torch, torch.nn as nn
bn = nn.BatchNorm1d(4)
x = torch.randn(2, 4)  # batch de 2 — estadísticas muy ruidosas en train mode
bn.train()
print(bn(x).var())     # varianza ~1 pero inestable
bn.eval()
print(bn(x).var())     # usa running_mean/running_var estables

La solución:

model.eval()           # ✓ BatchNorm usa running stats, Dropout desactivado
with torch.no_grad():  # ✓ no construye grafo computacional
    preds = model(val_data)
model.train()          # ✓ volver a train mode antes del siguiente epoch

Error 3: No validar dimensiones del input antes del forward

El problema:

model = MLPClassifier(input_dim=10, hidden_dims=[64], output_dim=3)
x_bad = torch.randn(8, 15)  # ❌ dimensión incorrecta
output = model(x_bad)       # RuntimeError críptico desde adentro de nn.Linear

Por qué falla:

El error que PyTorch lanza viene de las profundidades de at::mm — un stack trace de C++ que no te dice NADA sobre qué feature tenés mal, cuánto vale input_dim esperado ni dónde en tu pipeline se corrompió la dimensión.

La solución:

def forward(self, x: Tensor) -> Tensor:
    expected = self.layers[0][0].in_features  # ✓ validación explícita antes
    if x.shape[-1] != expected:
        raise ValueError(
            f"Input dim incorrecto: esperado {expected}, recibido {x.shape[-1]}. "
            f"¿Olvidaste el flatten o un reshape?"
        )
    return self._forward_impl(x)  # ✓ error descriptivo, no críptico

Error 4: Reutilizar tensores intermedios del forward sin detach()

El problema:

hidden_states = []

for batch in dataloader:
    out = model(batch)
    hidden_states.append(out)  # ❌ guarda el grafo computacional completo

loss = compute_loss(hidden_states)  # memoria GPU explota después de epoch 1

Por qué falla:

Cada tensor en hidden_states mantiene vivo todo el grafo computacional del batch correspondiente. Con N batches acumulados, la memoria crece linealmente.

La solución:

hidden_states = []

for batch in dataloader:
    with torch.no_grad():
        out = model(batch)
    hidden_states.append(out.detach().cpu())  # ✓ libera grafo, mueve a CPU

loss = compute_loss(torch.stack(hidden_states))  # ✓ sin OOM

🎯 Resumen Ejecutivo

Forward Propagation no es "pasar datos por la red". Es la evaluación determinista de:

$$\hat{\mathbf{y}} = g^{[L]}\!\left(\mathbf{W}^{[L]} \cdots g^{[1]}\!\left(\mathbf{W}^{[1]}\mathbf{x} + \mathbf{b}^{[1]}\right) \cdots + \mathbf{b}^{[L]}\right)$$

Materializada en PyTorch mediante:

$$\texttt{model}(\mathbf{x}) \xrightarrow{\text{construye}} \text{computation graph}$$

Con todas las implicaciones que eso conlleva:

  • El modo importa: model.train() vs model.eval() cambia el comportamiento de BatchNorm y Dropout — no es cosmético, cambia los resultados.
  • Los logits son sagrados: nunca apliqués softmax en el forward() si vas a usar CrossEntropyLoss — double softmax mata el gradiente.
  • El grafo tiene costo: fuera del training loop, torch.no_grad() no es opcional — es buenas prácticas y evita OOM en inferencia.

De una, hágale. Con esta base de Forward Propagation + PyTorch production-ready, ya sabés exactamente qué pasa dentro de model(x) — y lo más importante, sabés dónde buscar cuando algo falla. El Día 72 viene Backpropagation, que es cuando este grafo que acabamos de construir se recorre al revés.
