Día 29: One-Hot Encoding - Cuando las Categorías Necesitan Hablar el Idioma de las Matemáticas

Parcero, si alguna vez has intentado meter una columna de "ciudad" o "estado_civil" directo a un modelo de ML y te salió ese error de "ValueError: could not convert string to float", ya sabes de qué va esta vaina. Los algoritmos de machine learning no entienden texto; entienden números. Y ahí es donde One-Hot Encoding te salva el planning.

Hoy vamos al grano con sklearn.preprocessing.OneHotEncoder, la herramienta que convierte tus categorías en vectores binarios sin inventarse jerarquías donde no existen.

🎯 El Problema Real (Sin Humo)

Imaginate que tienes datos de empleados con la columna departamento: ['Ventas', 'TI', 'RRHH', 'Ventas', 'TI']. Si le metes eso crudo a un RandomForest, el modelo se va a inventar que TI > Ventas > RRHH solo porque internamente les asignó números [0, 1, 2]. Esa vaina es un bug conceptual brutal.

Analogía del Café ☕:

Es como si en la oficina dijeran "el tinto vale 1, el capuchino 2, el americano 3" y concluyeran que el americano es "mejor" que el tinto. No tiene sentido, ¿cierto? Son categorías independientes, no hay orden.

📐 Fundamento Matemático: Vectores Ortogonales

One-Hot convierte cada categoría en un vector de la base canónica de $\mathbb{R}^k$ (donde k es el número de categorías únicas). Esto garantiza ortogonalidad: ninguna categoría se "contamina" con otra.

Representación Formal

Dado un conjunto de categorías $C = \{c_1, c_2, \dots, c_k\}$, la función de encoding $\phi: C \to \{0,1\}^k$ se define como:

$$\phi(c_i) = \mathbf{e}_i = [0, \dots, 0, \underbrace{1}_{\text{posición } i}, 0, \dots, 0]$$

Paso a paso:

  1. $\mathbf{e}_i$ es el i-ésimo vector de la base canónica.
  2. Cada categoría $c_i$ se mapea a un vector donde solo la posición i tiene un 1.
  3. Todos los otros elementos son 0.

Propiedad Clave: Ortogonalidad

Para cualquier par de categorías distintas $c_i \neq c_j$:

$$\phi(c_i) \cdot \phi(c_j) = \mathbf{e}_i \cdot \mathbf{e}_j = 0$$

¿Qué significa esto en producción?

  • No hay correlación espuria entre categorías.
  • El modelo no puede "interpolar" entre Ventas y TI (no tiene sentido matemático).
  • Cada categoría contribuye de forma independiente a la predicción.

¿Por Qué No Simplemente Label Encoding?

Label Encoding asigna: Ventas → 0, TI → 1, RRHH → 2. Esto implica:

$$\text{TI} - \text{Ventas} = 1 - 0 = 1$$

El modelo aprende relaciones ordinales que no existen. Un árbol de decisión podría hacer split con departamento < 1.5, separando Ventas de {TI, RRHH}. Esa lógica es un bug silencioso.

Paso 2: Dataclass para Configuración

@dataclass(frozen=True)  # Inmutable = thread-safe
class OneHotConfig:
    """
    Configuración inmutable para OneHotEncoder.
    
    Attributes:
        handle_unknown: 'error' revienta en producción si ve categorías nuevas.
                       'ignore' crea un vector de ceros (útil para A/B testing).
        drop: 'first' para evitar multicolinealidad en regresión lineal.
              None si usas árboles (no les afecta).
        sparse_output: False para compatibilidad con pandas. True para RAM.
    """
    handle_unknown: str = 'ignore'  # Decisión arquitectónica clave
    drop: Optional[str] = None
    sparse_output: bool = False
    
    def __post_init__(self) -> None:
        """Validación en construcción, no en runtime."""
        valid_unknown = {'error', 'ignore', 'infrequent_if_exist'}
        if self.handle_unknown not in valid_unknown:
            raise ValueError(
                f"handle_unknown debe ser uno de {valid_unknown}, "
                f"recibido: {self.handle_unknown}"
            )

Decisiones de arquitectura:

  1. frozen=True: Esta config NO cambia. Si cambias parámetros, creas una nueva instancia. Evita bugs de mutación.
  2. handle_unknown='ignore': En producción, siempre van a llegar categorías que no viste en train. Si usas 'error', tu API se cae. Con 'ignore', creas un vector [0, 0, ..., 0] y el modelo lo maneja.
  3. drop='first': Solo si usas regresión lineal (multicolinealidad). Los árboles no lo necesitan.

Paso 3: Clase con Manejo de Errores Real

class CategoricalEncoder:
    """
    Wrapper production-ready de OneHotEncoder con validaciones.
    
    ¿Por qué no usar OneHotEncoder directo?
    - Manejo explícito de errores comunes (NaNs, tipos incorrectos).
    - Logging para debugging.
    - API consistente con pandas (devuelve DataFrames, no arrays).
    """
    
    __slots__ = ('_encoder', '_config', '_feature_names', '_is_fitted')
    # __slots__ reduce memoria ~40% vs dict-based attributes
    
    def __init__(self, config: OneHotConfig) -> None:
        self._config = config
        self._encoder = OneHotEncoder(
            handle_unknown=config.handle_unknown,
            drop=config.drop,
            sparse_output=config.sparse_output
        )
        self._feature_names: Optional[List[str]] = None
        self._is_fitted: bool = False
        logger.info(f"CategoricalEncoder inicializado con config: {config}")
    
    def fit(self, X: pd.DataFrame, columns: List[str]) -> 'CategoricalEncoder':
        """
        Aprende las categorías únicas de las columnas especificadas.
        
        Args:
            X: DataFrame con datos de entrenamiento
            columns: Lista de columnas categóricas a encodear
            
        Returns:
            self (patrón builder para chaining)
            
        Raises:
            ValueError: Si las columnas no existen o tienen NaNs
        """
        self._validate_input(X, columns)
        
        try:
            X_cat = X[columns].copy()
            self._encoder.fit(X_cat)
            
            # Guardar nombres de features para interpretabilidad
            self._feature_names = self._encoder.get_feature_names_out(columns).tolist()
            self._is_fitted = True
            
            logger.info(
                f"Encoder fitted. Categorías encontradas: "
                f"{sum(len(cats) for cats in self._encoder.categories_)}"
            )
            
        except Exception as e:
            logger.error(f"Error en fit: {e}")
            raise
        
        return self
    
    def transform(self, X: pd.DataFrame, columns: List[str]) -> pd.DataFrame:
        """
        Transforma categorías a one-hot vectors.
        
        Returns:
            DataFrame con columnas originales no-categóricas + one-hot features
        """
        if not self._is_fitted:
            raise RuntimeError(
                "Encoder no está fitted. Llamá .fit() primero, parcero."
            )
        
        self._validate_input(X, columns)
        
        try:
            X_encoded = X.copy()
            X_cat = X_encoded[columns]
            
            # Transformar y convertir a DataFrame
            encoded_array = self._encoder.transform(X_cat)
            encoded_df = pd.DataFrame(
                encoded_array,
                columns=self._feature_names,
                index=X.index  # Preservar índice original
            )
            
            # Reemplazar columnas categóricas con one-hot
            X_encoded = X_encoded.drop(columns=columns)
            X_encoded = pd.concat([X_encoded, encoded_df], axis=1)
            
            logger.info(f"Transform exitoso. Shape: {X_encoded.shape}")
            return X_encoded
            
        except Exception as e:
            logger.error(f"Error en transform: {e}")
            raise
    
    def _validate_input(self, X: pd.DataFrame, columns: List[str]) -> None:
        """
        Validaciones para evitar errores silenciosos en producción.
        """
        # Check 1: Tipo correcto
        if not isinstance(X, pd.DataFrame):
            raise TypeError(
                f"X debe ser DataFrame, recibido: {type(X).__name__}"
            )
        
        # Check 2: Columnas existen
        missing_cols = set(columns) - set(X.columns)
        if missing_cols:
            raise ValueError(
                f"Columnas faltantes en X: {missing_cols}"
            )
        
        # Check 3: No NaNs (one-hot no los maneja bien)
        nan_cols = [col for col in columns if X[col].isna().any()]
        if nan_cols:
            raise ValueError(
                f"Columnas con NaNs detectados: {nan_cols}. "
                f"Imputá primero con SimpleImputer(strategy='most_frequent')"
            )
    
    def get_feature_names(self) -> List[str]:
        """Para inspección y feature importance."""
        if not self._is_fitted:
            raise RuntimeError("Encoder no fitted")
        return self._feature_names.copy()

¿Qué evita este código?

  1. __slots__: Reduce uso de memoria. En un DataFrame con 1M filas, esto importa.
  2. Validación de NaNs: OneHotEncoder NO maneja NaNs. Si no checkeas, el error aparece 3 funciones después y perdés 20 minutos debuggeando.
  3. Preservar índice: Si estás en un pipeline con time series, perder el índice te rompe todo.
  4. Logging: Cuando algo falla en producción a las 3 AM, necesitas logs, no prints.

Paso 4: Ejemplo Completo con Pipeline

def demo_one_hot_encoding() -> None:
    """
    Ejemplo realista: encoding de datos de empleados.
    """
    # Datos sintéticos (en producción vienen de SQL/S3)
    data = pd.DataFrame({
        'empleado_id': [1, 2, 3, 4, 5],
        'departamento': ['TI', 'Ventas', 'TI', 'RRHH', 'Ventas'],
        'ciudad': ['Bogotá', 'Medellín', 'Bogotá', 'Cali', 'Medellín'],
        'salario': [5000, 3500, 5500, 4000, 3800]
    })
    
    print("DataFrame original:")
    print(data)
    print("\n" + "="*60 + "\n")
    
    # Config para producción
    config = OneHotConfig(
        handle_unknown='ignore',  # CRÍTICO para nuevas ciudades
        drop=None,                # Usaremos RandomForest (no regresión)
        sparse_output=False
    )
    
    # Encoder
    encoder = CategoricalEncoder(config)
    categorical_cols = ['departamento', 'ciudad']
    
    # Fit en train (asumiendo estos son los datos de train)
    encoder.fit(data, categorical_cols)
    
    # Transform
    data_encoded = encoder.transform(data, categorical_cols)
    
    print("DataFrame después de One-Hot Encoding:")
    print(data_encoded)
    print(f"\nShape original: {data.shape}")
    print(f"Shape encoded: {data_encoded.shape}")
    print(f"\nFeatures generados:")
    for i, name in enumerate(encoder.get_feature_names(), 1):
        print(f"  {i}. {name}")
    
    # Simular datos de producción con categoría nueva
    print("\n" + "="*60)
    print("CASO PRODUCCIÓN: Nueva ciudad (Barranquilla) 🚨\n")
    
    new_data = pd.DataFrame({
        'empleado_id': [6],
        'departamento': ['TI'],
        'ciudad': ['Barranquilla'],  # NO estaba en train
        'salario': [4500]
    })
    
    try:
        new_encoded = encoder.transform(new_data, categorical_cols)
        print("Transform exitoso con handle_unknown='ignore':")
        print(new_encoded)
        print("\n⚠️  Nota: Las columnas de 'ciudad' son [0, 0, 0]")
        print("El modelo lo maneja como 'ciudad desconocida'")
    except Exception as e:
        print(f"Error: {e}")


if __name__ == "__main__":
    demo_one_hot_encoding()

Paso 5: Integración con Pipeline de sklearn

from sklearn.pipeline import Pipeline
from sklearn.ensemble import RandomForestClassifier
from sklearn.model_selection import train_test_split

def create_ml_pipeline() -> Pipeline:
    """
    Pipeline completo con preprocessing automático.
    
    ¿Por qué ColumnTransformer?
    - Aplica transformaciones solo a columnas específicas.
    - Maneja features numéricas y categóricas en un solo paso.
    - Previene data leakage (fit solo en train).
    """
    
    # Definir qué hacer con cada tipo de columna
    categorical_features = ['departamento', 'ciudad']
    numeric_features = ['salario']
    
    # ColumnTransformer: la manera correcta de hacer preprocessing
    preprocessor = ColumnTransformer(
        transformers=[
            ('cat', OneHotEncoder(
                handle_unknown='ignore',
                sparse_output=False
            ), categorical_features),
            ('num', 'passthrough', numeric_features)  # Dejar numéricas sin cambios
        ],
        remainder='drop'  # Ignorar otras columnas (ej: empleado_id)
    )
    
    # Pipeline completo
    pipeline = Pipeline(steps=[
        ('preprocessor', preprocessor),
        ('classifier', RandomForestClassifier(n_estimators=100, random_state=42))
    ])
    
    return pipeline

⚠️ Trampas Comunes (Y Cómo NO Quedar Embalado)

1. Explosión de Dimensionalidad

Si tenés una columna user_id con 50,000 usuarios únicos, One-Hot crea 50,000 columnas. Esa vaina revienta la RAM.

Solución:

  • Para high-cardinality features, usá Target Encoding o Embeddings (día 30+).
  • Regla: One-Hot solo para categorías con k < 50 valores únicos.
# Check antes de encodear
high_cardinality = [
    col for col in categorical_cols 
    if df[col].nunique() > 50
]
if high_cardinality:
    logger.warning(f"High cardinality detected: {high_cardinality}")

2. Multicolinealidad en Regresión Lineal

Si tenés k categorías, One-Hot crea k columnas. Pero $\mathbf{e}_1 + \mathbf{e}_2 + \dots + \mathbf{e}_k = [1, 1, \dots, 1]$(el vector de unos). Esto rompe la invertibilidad de $X^TX$ en regresión.

Solución matemática:

$$\text{rank}(X) = k - 1 \implies \text{usar } \texttt{drop='first'}$$

Dropeas una categoría (la "referencia") y las otras se interpretan relativo a ella.

3. Data Leakage con fit_transform en Test

# ❌ MAL - Leakage
encoder.fit_transform(X_test)  # Test "ve" sus propias categorías

# ✅ BIEN
encoder.fit(X_train)
encoder.transform(X_test)

Si fitteas en test, el modelo "sabe" qué categorías van a venir. Eso infla las métricas artificialmente.

🎯 Conclusión: Por Qué Este Código Importa

One-Hot Encoding parece trivial, pero en producción es donde más bugs silenciosos aparecen:

  • NaNs no manejados → ValueError en runtime.
  • Categorías nuevas → Model crash en predicción.
  • High cardinality → OOM (Out of Memory).

Este código te da:

✅ Validaciones explícitas que fallan rápido (no 3 funciones después).

✅ Logging para debugging post-mortem.

✅ Inmutabilidad (config frozen) para thread-safety.

✅ Typing estricto para que tu IDE te ayude.

En resumen, parcero: One-Hot no es solo "convertir texto a números". Es diseñar una representación matemáticamente correcta que tu modelo pueda aprender sin inventarse jerarquías espurias. Y en producción, es manejar los edge cases antes de que te despierten a las 3 AM. 🔥
