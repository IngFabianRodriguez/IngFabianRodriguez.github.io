La Entropía de Shannon o "¿Qué tanto desorden hay en mi sistema?"

1. Introducción: El Ruido, la Señal y el "Susto"

¡Qué hubo, equipo! Sírvanse ese tinto bien oscuro, ojalá sin azúcar para que despierten las neuronas.

Hoy vamos a hablar de un tema que separa a los que "escriben código" de los que "diseñan sistemas resilientes". Vamos a hablar de la Entropía de Shannon. Y no, no se me asusten ni cierren la pestaña pensando que esto es una clase de física teórica de las que uno pasaba raspando en la universidad.

Mire, le voy a ser sincero. Llevo más de 10 años en esta "trinchera", viendo logs pasar como en la Matrix. Y si hay algo que he aprendido a las malas —a punta de incidentes un domingo en la tarde— es que el silencio en los logs no significa que todo esté bien.

A veces, su API devuelve 200 OK y usted cree que es un genio. Pero resulta que está devolviendo 200 OK con un cuerpo JSON vacío. O resulta que el usuario está haciendo click en el mismo botón 500 veces por segundo. El volumen de tráfico parece normal, los códigos de estado parecen normales, pero la información ha cambiado.

Ahí entra la Entropía. Es el termómetro que mide la sorpresa.

¿Su sistema se comporta como un relojito suizo (Entropía 0) o como una plaza de mercado un sábado a mediodía (Entropía Máxima)? Hoy vamos a desglosar esto con bisturí, vamos a entender cada garabato de la fórmula y vamos a montar un script que pille anomalías que Datadog o New Relic por defecto no ven.

¡Hágale, que hoy hay clase magistral!

2. Fundamentos Matemáticos: La Anatomía de la Incertidumbre

Vamos a quitarnos el miedo a la matemática. La matemática no muerde, lo que muerde es un bug en producción por no saber estadística.

2.1. El Concepto en "Criollo" (Analogía de la Chismosa)

Imagínese que usted tiene una tía chismosa en el barrio.

  • Escenario A (Baja Entropía - Cero Sorpresa):
    Usted se encuentra a la tía y le dice: "Mijo, ¿sabía que mañana va a salir el sol?".
    Usted queda igual. Esa frase tiene CERO información nueva. Usted ya sabía eso con 100% de probabilidad.
    Información ganada = 0 bits.
  • Escenario B (Alta Entropía - Alta Sorpresa):
    La tía le dice: "Mijo, ¿sabía que al vecino, el que es cura, se ganó el Baloto y se va a casar con Shakira?".
    ¡Pum! Su cerebro explota. Es un evento con BAJÍSIMA probabilidad.
    Información ganada = Muchos bits.

Conclusión de Arquitecto:
La información es inversamente proporcional a la probabilidad.

  • Lo muy probable (el sistema funcionando bien) tiene poca información/entropía.
  • Lo improbable (un error raro, un ataque nuevo) tiene mucha información/entropía.

2.2. La Fórmula Descuartizada (Paso a Paso)

Claude Shannon, que era un genio absoluto, nos dejó esta joya. No la miren por encima, léanla conmigo:

$$
H(X) = - \sum_{i=1}^{n} P(x_i) \cdot \log_2 P(x_i)
$$

¿Qué diablos significa cada símbolo? Vamos a "desbugear" la fórmula:

  1. $X$ (La Variable): Es lo que estamos midiendo. Digamos que son los "Códigos de Error HTTP" en sus logs (200, 404, 500).
  2. $P(x_i)$ (La Probabilidad): Es la frecuencia de cada evento.
    • Si de 100 requests, 90 son 200 OK, entonces $P(200) = 0.9$.
    • Si 10 son 500 Error, entonces $P(500) = 0.1$.
  3. $\log_2 P(x_i)$ (El Medidor de Sorpresa): Aquí está la magia.
    • El logaritmo de 1 (certeza total) es 0. Si algo siempre pasa, sorpresas = 0.
    • El logaritmo de un número pequeño (ej. 0.01) es un número negativo grande.
    • ¿Por qué logaritmo? Porque nos permite sumar "bits". Es la forma matemática de preguntar "¿cuántas preguntas de Sí/No necesito hacer para adivinar qué pasó?".
  4. El Signo Menos ($-$) al inicio:
    • Como el $\log$ de una probabilidad (que es menor a 1) siempre da negativo, el resultado sería negativo.
    • Como no existe la "información negativa" (a menos que usted vea noticias falsas, jaja), le ponemos un menos afuera para volverlo positivo.
  5. $\sum$ (La Sumatoria):
    • No nos sirve saber la sorpresa de un solo evento. Queremos el promedio de sorpresa de todo el sistema. Por eso sumamos todo ponderado por su probabilidad.

En Resumen:

La Entropía es el Promedio de Sorpresa que usted se lleva cada vez que mira un log de su sistema.

3. Implementación "Production-Ready": El Código

Suficiente carreta teórica. Vamos al código. Como Arquitecto, exijo que esto corra rápido y no se caiga. Usaremos scipy porque está optimizado en C y Fortran por debajo. No intente hacer esto con for loops en Python puro si tiene millones de datos, o se va a quedar esperando hasta el 2030.

El Reto Técnico

Vamos a crear un Monitor de Entropía. Le vamos a pasar una lista de eventos (digamos, User-Agents de un servidor web) y nos tiene que decir si estamos bajo un ataque de botnet (Alta Entropía) o si es tráfico normal.

import logging
import time
import math
import numpy as np
from scipy import stats
from typing import List, Optional, Protocol
from dataclasses import dataclass
from collections import Counter
from functools import wraps

# --- 1. Configuración de Logging ---
# Un log sin timestamp y nivel es basura.
logging.basicConfig(
    level=logging.INFO,
    format='%(asctime)s - %(name)s - %(levelname)s - %(message)s'
)
logger = logging.getLogger("EntropyGuard")

# --- 2. Definición de Estructuras de Datos (DTOs) ---
@dataclass(slots=True) # slots=True ahorra RAM, clave en Big Data
class EntropyReport:
    """
    DTO (Data Transfer Object) inmutable para transportar los resultados.
    """
    value: float              # El valor crudo de la entropía
    unit: str                 # 'bits' o 'nats'
    sample_size: int          # Cuántos datos analizamos
    normalized_score: float   # De 0.0 a 1.0 (Qué tan lleno está el 'vaso' de caos)
    is_anomaly: bool          # El veredicto final

# --- 3. Decoradores de Calidad ---
def production_guard(func):
    """
    Este decorador es su chaleco antibalas.
    Mide tiempo y atrapa errores para que el proceso principal no muera.
    """
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        func_name = func.__name__
        try:
            result = func(*args, **kwargs)
            elapsed = (time.perf_counter() - start) * 1000

            # SLA Check: Si el cálculo toma más de 50ms, warn.
            if elapsed > 50:
                logger.warning(f"⚠️ {func_name} está lento: {elapsed:.2f}ms. Revise el tamaño del batch.")
            return result

        except Exception as e:
            # En producción, logueamos el error y relanzamos o manejamos graceful degradation
            logger.critical(f"🔥 Pánico en {func_name}: {str(e)}")
            raise e
    return wrapper

# --- 4. La Lógica del Negocio (Core) ---
class ShannonMonitor:
    """
    Clase especializada en medir el desorden de una señal discreta.
    """

    # __slots__ define explícitamente los atributos.
    # Python no crea el diccionario __dict__, ahorrando ~40% de RAM por instancia.
    __slots__ = ('_threshold', '_base_log')

    def __init__(self, anomaly_threshold: float = 0.8, base: int = 2):
        """
        Args:
            anomaly_threshold: Si la entropía normalizada supera esto (0-1), es anomalía.
            base: 2 para bits (informática), 'e' para física/natural.
        """
        self._threshold = anomaly_threshold
        self._base_log = base

    def _get_probabilities(self, data: List[str]) -> np.ndarray:
        """
        Convierte una lista de strings crudos en un vector numérico de probabilidades.

        Ejemplo: ['A', 'A', 'B'] -> [0.66, 0.33]
        """
        if not data:
            return np.array([])

        # Counter es C-optimized en Python moderno, muy rápido.
        counts = Counter(data)
        total = sum(counts.values())

        # Convertimos a numpy array para vectorización matemática
        probs = np.array(list(counts.values()), dtype=np.float64) / total
        return probs

    @production_guard
    def analyze_batch(self, batch: List[str]) -> EntropyReport:
        """
        El corazón del sistema. Calcula H(X) y normaliza.
        """
        # Validación de entrada (Fail fast)
        if not batch:
            logger.warning("Batch vacío recibido. Entropía indefinida (0).")
            return EntropyReport(0.0, "bits", 0, 0.0, False)

        # 1. Obtener distribución P(x)
        pk = self._get_probabilities(batch)

        # 2. Calcular Entropía de Shannon
        # stats.entropy hace la sumatoria y el logaritmo internamente.
        # Es mucho más estable numéricamente que hacerlo a mano (evita underflows).
        entropy_val = stats.entropy(pk, base=self._base_log)

        # 3. Calcular la Entropía Máxima Posible (H_max)
        # La entropía es máxima cuando TODOS los eventos son igual de probables (Distribución Uniforme).
        # H_max = log(N_categorias)
        n_unique_categories = len(pk)

        if n_unique_categories <= 1:
            # Si solo hay 1 categoría (todo es igual), la entropía es 0.
            max_entropy = 0.0
            normalized = 0.0
        else:
            max_entropy = math.log(n_unique_categories, self._base_log)
            # Normalizamos de 0 a 1 para tener un "Score de Caos" agnóstico a la escala
            normalized = entropy_val / max_entropy

        # 4. Detección de Anomalía
        # Si el score normalizado supera el umbral, estamos en territorio de caos/ataque.
        is_anomaly = normalized > self._threshold

        logger.info(
            f"📊 Análisis completado: H={entropy_val:.3f} | "
            f"Score={normalized:.2f} | Categorías={n_unique_categories} | Anomaly={is_anomaly}"
        )

        return EntropyReport(
            value=entropy_val,
            unit="bits" if self._base_log == 2 else "nats",
            sample_size=len(batch),
            normalized_score=normalized,
            is_anomaly=is_anomaly
        )

# --- 5. Simulación (Mocks) ---
# Porque código sin probar es código legado instantáneo.

def run_simulation():
    print("\\n--- 🔬 Iniciando Laboratorio de Entropía ---")
    monitor = ShannonMonitor(anomaly_threshold=0.9) # Umbral alto, solo alertar caos total

    # CASO 1: Tráfico Normal (Predecible)
    # La mayoría de usuarios usa Chrome o Firefox. Pocos usan cosas raras.
    # Distribución sesgada -> Baja Entropía.
    normal_traffic = (
        ["Chrome"] * 700 +
        ["Safari"] * 200 +
        ["Firefox"] * 90 +
        ["HackerBot"] * 10
    )
    print(f"\\n1️⃣ Escenario: Día tranquilo en la oficina.")
    report1 = monitor.analyze_batch(normal_traffic)
    print(f"   ➤ Entropía: {report1.value:.3f} bits")
    print(f"   ➤ Nivel de Caos: {report1.normalized_score:.1%}")
    print(f"   ➤ ¿Debo preocuparme?: {'SI 🚨' if report1.is_anomaly else 'Todo bien 👍'}")

    # CASO 2: Ataque de Botnet (Caos Aleatorio)
    # Los atacantes rotan el User-Agent en cada petición para evadir bloqueos.
    # Distribución Uniforme -> Máxima Entropía.
    attack_traffic = [f"BotAgent_{i}" for i in range(1000)] # 1000 agentes únicos

    print(f"\\n2️⃣ Escenario: Ataque distribuido (Randomized User-Agents).")
    report2 = monitor.analyze_batch(attack_traffic)
    print(f"   ➤ Entropía: {report2.value:.3f} bits")
    # Nota: log2(1000) ≈ 9.96 bits. Debería dar muy cerca.
    print(f"   ➤ Nivel de Caos: {report2.normalized_score:.1%}")
    print(f"   ➤ ¿Debo preocuparme?: {'SI 🚨' if report2.is_anomaly else 'Todo bien 👍'}")

if __name__ == "__main__":
    run_simulation()

4. Historia de Guerra: El Incidente del "Silencio Ruidoso"

Déjeme contarle una anécdota rápida para que vea que esto no es teoría.

Hace unos años, en una Fintech, tenían un dashboard precioso. Todo verde. Tasa de errores 5XX por debajo del 0.1%. Latencia perfecta.

Pero soporte al cliente estaba estallado. "No puedo pagar", decían los usuarios.

¿Qué pasó?

Resulta que habían desplegado un bug en el Frontend que, en lugar de generar un error 500 en el backend, enviaba una petición mala que el backend respondía con un 200 OK pero con un mensaje interno de "Invalid Operation".

Para sus métricas tradicionales (contar errores 500), el sistema estaba perfecto.
Pero si hubiéran medido la Entropía de los logs de respuesta, habrían visto algo raro.

Normalmente, los logs de respuesta varían mucho (IDs de transacción, montos, fechas).
Ese día, el 90% de los logs eran idénticos: {"status": "ok", "msg": "invalid"}.
La entropía del sistema colapsó (bajó casi a cero).

Si hubiera tenido este script corriendo, la alerta no habría sido "Demasiados Errores", sino "Demasiada Uniformidad".

Lección de Mentor:
La entropía te avisa cuando hay mucho desorden (ataque) O cuando hay demasiado orden (bug sistémico). Los extremos son malos. La vida, y el software sano, vive en el medio.
