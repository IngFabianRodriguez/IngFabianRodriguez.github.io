Día 30: Train/Test Split — Por Qué Tu Modelo "Genial" en Desarrollo Se Cae en Producción 🎯

Parcero, hablemos de una de las vainas más básicas pero que más daño hacen cuando las ignoras: separar datos de entrenamiento y prueba. Suena obvio, ¿cierto? Pero he visto proyectos completos irse al piso porque alguien pensó que "evaluar con los mismos datos de entrenamiento" era suficiente. Spoiler: no lo es.

El Problema Real (El Que Te Quita el Sueño)

Imaginate que estás preparando un parcial. Te memorizas el quiz de práctica palabra por palabra, sacas 100/100 en ese quiz... pero cuando llega el examen real con preguntas diferentes, te raja. Eso es exactamente lo que pasa con tu modelo cuando no separas los datos correctamente.

En producción, tu modelo va a ver datos que nunca ha visto antes. Si solo lo evaluás con datos que ya conoce, estás midiendo memorización, no generalización. Y en ML, la memorización sin generalización es overfitting puro y duro.

La Analogía del Restaurante

Pensá en esto: sos el dueño de un restaurante y querés saber si tu nuevo chef es bueno. ¿Qué hacés?

  • Opción A (MAL): Le pedís que prepare los mismos 10 platos que practicó toda la semana, probás esos, y decís "es un crack". Llega el sábado con clientes reales pidiendo platos diferentes → desastre.
  • Opción B (BIEN): Separás las recetas. Le das 80% para que practique, y el 20% restante lo guardás para probar si realmente aprendió a cocinar o solo memorizó esas recetas específicas.

Esa vaina tan simple es train/test split.

La Matemática Detrás (Por Qué Funciona)

Asumamos que tenés un dataset $\mathcal{D}$ con $N$ muestras:

$$\mathcal{D} = \{(\mathbf{x}_1, y_1), (\mathbf{x}_2, y_2), \ldots, (\mathbf{x}_N, y_N)\}$$

Donde $\mathbf{x}_i \in \mathbb{R}^d$ son las features y $y_i$ es el target.

División Train/Test

Dividimos $\mathcal{D}$ en dos subconjuntos disjuntos:

$$\mathcal{D}_{\text{train}} \cap \mathcal{D}_{\text{test}} = \emptyset$$

$$\mathcal{D}_{\text{train}} \cup \mathcal{D}_{\text{test}} = \mathcal{D}$$

Típicamente con proporción:

$$|\mathcal{D}_{\text{train}}| = (1 - \alpha) \cdot N, \quad |\mathcal{D}_{\text{test}}| = \alpha \cdot N$$

Donde $\alpha \in [0.1, 0.3]$ comúnmente (20-30% para test es estándar).

El Objetivo: Minimizar el Error de Generalización

Lo que realmente nos importa es el error esperado en datos no vistos:

$$\mathcal{L}_{\text{true}}(\theta) = \mathbb{E}_{(\mathbf{x}, y) \sim P_{\text{data}}} \left[ \ell(f_\theta(\mathbf{x}), y) \right]$$

Desglose paso a paso:

  1. $P_{\text{data}}$ La distribución real (y desconocida) de donde vienen los datos
  2. $f_\theta$: Nuestro modelo con parámetros $\theta$
  3. $\ell(\cdot, \cdot)$: Función de pérdida (MSE, cross-entropy, etc.)
  4. $\mathbb{E}[\cdot]$: El valor esperado (promedio teórico sobre TODOS los posibles datos)

El problema: No conocemos $P_{\text{data}}$, solo tenemos una muestra finita $\mathcal{D}$.

Aproximación Empírica

Lo que sí podemos calcular es el error empírico en nuestro test set:

$$\hat{\mathcal{L}}_{\text{test}}(\theta) = \frac{1}{|\mathcal{D}_{\text{test}}|} \sum_{(\mathbf{x}, y) \in \mathcal{D}_{\text{test}}} \ell(f_\theta(\mathbf{x}), y)$$

Por qué funciona:

Por la Ley de Grandes Números, si $\mathcal{D}_{\text{test}}$ es una muestra i.i.d. (independiente e idénticamente distribuida) de $P_{\text{data}}$:

$$\hat{\mathcal{L}}_{\text{test}}(\theta) \xrightarrow{N \to \infty} \mathcal{L}_{\text{true}}(\theta)$$

Es decir, el error en el test set converge al error real que veríamos en producción.

Descomposición Sesgo-Varianza

El error de generalización se puede descomponer:

$$\mathbb{E}[\mathcal{L}_{\text{test}}] = \text{Bias}^2 + \text{Variance} + \text{Irreducible Error}$$

Explicación:

  1. Bias (Sesgo): Qué tanto se equivoca tu modelo en promedio. Un modelo muy simple (como regresión lineal para datos no lineales) tiene alto sesgo.
  2. Variance (Varianza): Qué tanto cambia tu modelo si entrenás con diferentes subconjuntos de datos. Modelos muy complejos (como árboles profundos) tienen alta varianza.
  3. Irreducible Error: Ruido inherente en los datos que ningún modelo puede capturar.

El train/test split te ayuda a detectar:

  • Alto sesgo: Mal desempeño tanto en train como en test
  • Alta varianza (overfitting): Excelente en train, horrible en test

Ando embalado con implementaciones que fallen silenciosamente. Acá va un módulo completo que maneja todos los edge cases que te vas a encontrar en producción.

train_test_split_module.py

Módulo robusto para división de datos con validaciones exhaustivas.

from typing import (
    TypeVar, Generic, Protocol, Tuple, Optional, 
    Union, List, cast, Any
)
from dataclasses import dataclass
import numpy as np
import numpy.typing as npt
from enum import Enum
import logging
from sklearn.model_selection import train_test_split as sklearn_split

logging.basicConfig(

    format='%(asctime)s - %(name)s - %(levelname)s - %(message)s'
)

class StratifyStrategy(Enum):
    """
    Estrategias de estratificación.
    
    Arquitectura: Enum en vez de strings para type safety y autocomplete.
    """
    NONE = "none"
    AUTO = "auto"
    MANUAL = "manual"


class ShuffleMode(Enum):
    """Modos de shuffle con semántica clara."""
    RANDOM = "random"
    DETERMINISTIC = "deterministic"
    NO_SHUFFLE = "no_shuffle"

Por qué estos tipos:

  • Enum en vez de strings: Evita typos y mejora autocomplete
  • Protocol: Permite trabajar con arrays de NumPy, Pandas, o cualquier cosa array-like sin acoplar
  • Logging configurado: Cuando algo falle en prod a las 3am, vas a agradecer los logs

Paso 2: Validaciones y Excepciones Custom

class DataSplitError(Exception):
    """
    Excepción base para errores en split de datos.
    
    Decisión: Jerarquía de excepciones custom en vez de usar ValueError genérico.
    Razón: Permite catch selectivo y mensajes de error informativos.
    """
    pass


class InsufficientDataError(DataSplitError):
    """No hay suficientes datos para hacer el split."""
    pass


class InvalidProportionError(DataSplitError):
    """La proporción de split es inválida."""
    pass


class StratificationError(DataSplitError):
    """Error al intentar estratificar."""
    pass


@dataclass(frozen=True, slots=True)
class SplitConfig:
    """
    Configuración inmutable para train/test split.
    
    Arquitectura:
    - frozen=True: Immutability para evitar modificaciones accidentales
    - slots=True: Reduce uso de memoria (~40% menos por instancia)
    """
    test_size: float = 0.2
    random_state: Optional[int] = 42
    shuffle: bool = True
    stratify_strategy: StratifyStrategy = StratifyStrategy.NONE
    min_samples_per_class: int = 2
    
    def __post_init__(self) -> None:
        """Validación de configuración al construir."""
        if not 0.0 < self.test_size < 1.0:
            raise InvalidProportionError(
                f"test_size debe estar en (0, 1), recibido: {self.test_size}"
            )
        
        if self.min_samples_per_class < 1:
            raise InvalidProportionError(
                f"min_samples_per_class debe ser >= 1, "
                f"recibido: {self.min_samples_per_class}"
            )
        
        logger.info(
            f"SplitConfig creado: test_size={self.test_size}, "
            f"random_state={self.random_state}, shuffle={self.shuffle}"
        )

Decisión clave:

  • frozen=True: Si alguien intenta modificar la config después de crear el split, explota inmediatamente. Nada de bugs silenciosos.
  • __slots__: En producción con millones de configuraciones, ahorrar 40% de memoria no es negociable.

Paso 3: Clase Principal con Toda la Lógica

class TrainTestSplitter(Generic[npt.NDArray[Any]]):
    """
    Splitter robusto con validaciones exhaustivas y logging.
    
    Arquitectura:
    - Generic para soportar diferentes tipos de arrays
    - Métodos privados para separación de responsabilidades
    - Validación en cada paso crítico
    """
    
    __slots__ = ('_config', '_n_samples', '_n_features')
    
    def __init__(self, config: Optional[SplitConfig] = None) -> None:
        """
        Args:
            config: Configuración del split. Si es None, usa defaults.
        """
        self._config = config or SplitConfig()
        self._n_samples: int = 0
        self._n_features: int = 0
        logger.info("TrainTestSplitter inicializado")
    
    def _validate_inputs(
        self,
        X: npt.NDArray[Any],
        y: Optional[npt.NDArray[Any]] = None
    ) -> None:
        """
        Validación exhaustiva de inputs.
        
        Decisión: Fail fast. Mejor explotar acá que en mitad del entrenamiento.
        """
        if X.size == 0:
            raise InsufficientDataError("X está vacío")
        
        if X.ndim < 1:
            raise ValueError(f"X debe tener al menos 1 dimensión, tiene: {X.ndim}")
        
        self._n_samples = X.shape[0]
        self._n_features = X.shape[1] if X.ndim > 1 else 1
        
        # Validar que hay suficientes muestras para el split
        min_samples_needed = int(
            max(
                2,  # Mínimo absoluto
                self._config.min_samples_per_class / self._config.test_size
            )
        )
        
        if self._n_samples < min_samples_needed:
            raise InsufficientDataError(
                f"Se necesitan al menos {min_samples_needed} muestras "
                f"para test_size={self._config.test_size}. "
                f"Recibido: {self._n_samples}"
            )
        
        # Si hay y, validar consistencia
        if y is not None:
            if len(y) != self._n_samples:
                raise ValueError(
                    f"X y y tienen diferentes números de muestras: "
                    f"{self._n_samples} vs {len(y)}"
                )
        
        logger.debug(
            f"Validación exitosa: {self._n_samples} muestras, "
            f"{self._n_features} features"
        )
    
    def _check_stratification_feasibility(
        self,
        y: npt.NDArray[Any]
    ) -> bool:
        """
        Verifica si la estratificación es posible.
        
        Retorna:
            True si es posible estratificar, False otherwise.
        
        Razón: sklearn.train_test_split falla si hay clases con muy pocas muestras.
        Mejor detectarlo antes y loggear una advertencia.
        """
        unique, counts = np.unique(y, return_counts=True)
        
        min_count = counts.min()
        test_samples_per_class = min_count * self._config.test_size
        
        if test_samples_per_class < 1:
            logger.warning(
                f"Estratificación imposible: clase con menor frecuencia tiene "
                f"{min_count} muestras, lo que resulta en "
                f"{test_samples_per_class:.2f} muestras en test set"
            )
            return False
        
        if min_count < self._config.min_samples_per_class:
            logger.warning(
                f"Clase con {min_count} muestras está por debajo del mínimo "
                f"configurado ({self._config.min_samples_per_class})"
            )
            return False
        
        return True
    
    def split(
        self,
        X: npt.NDArray[Any],
        y: Optional[npt.NDArray[Any]] = None
    ) -> Tuple[
        npt.NDArray[Any], 
        npt.NDArray[Any],
        Optional[npt.NDArray[Any]],
        Optional[npt.NDArray[Any]]
    ]:
        """
        Divide datos en train y test sets.
        
        Args:
            X: Features (samples, features)
            y: Targets (samples,) - opcional
        
        Returns:
            (X_train, X_test, y_train, y_test)
            Si y es None, retorna (X_train, X_test, None, None)
        
        Raises:
            DataSplitError: Si el split no es posible
        """
        self._validate_inputs(X, y)
        
        # Determinar si estratificar
        stratify_array: Optional[npt.NDArray[Any]] = None
        
        if self._config.stratify_strategy == StratifyStrategy.AUTO and y is not None:
            # Auto: intentar estratificar si es posible
            if self._check_stratification_feasibility(y):
                stratify_array = y
                logger.info("Estratificación automática activada")
            else:
                logger.info("Estratificación automática desactivada (no factible)")
        
        elif self._config.stratify_strategy == StratifyStrategy.MANUAL:
            if y is None:
                raise StratificationError(
                    "Estratificación manual requiere y != None"
                )
            if not self._check_stratification_feasibility(y):
                raise StratificationError(
                    "Estratificación manual solicitada pero no es factible"
                )
            stratify_array = y
            logger.info("Estratificación manual activada")
        
        # Ejecutar el split
        try:
            if y is not None:
                X_train, X_test, y_train, y_test = sklearn_split(
                    X, y,
                    test_size=self._config.test_size,
                    random_state=self._config.random_state,
                    shuffle=self._config.shuffle,
                    stratify=stratify_array
                )
            else:
                X_train, X_test = sklearn_split(
                    X,
                    test_size=self._config.test_size,
                    random_state=self._config.random_state,
                    shuffle=self._config.shuffle
                )
                y_train, y_test = None, None
            
            logger.info(
                f"Split exitoso: Train={len(X_train)} samples, "
                f"Test={len(X_test)} samples"
            )
            
            return X_train, X_test, y_train, y_test
        
        except ValueError as e:
            raise DataSplitError(f"Error en sklearn split: {str(e)}") from e
    
    @property
    def config(self) -> SplitConfig:
        """Retorna configuración actual (inmutable)."""
        return self._config

Decisiones arquitectónicas críticas:

  1. Validación en múltiples niveles: Input → Factibilidad → Ejecución
  2. Logging estratégico: Info en operaciones exitosas, Warning en casos edge, Debug para troubleshooting
  3. Type hints estrictos: El IDE te avisa si estás pasando tipos incorrectos
  4. Slots en toda la jerarquía: Ahorro de memoria escalable

Paso 4: Funciones de Utilidad y Análisis

def analyze_split_distribution(
    y_train: npt.NDArray[Any],
    y_test: npt.NDArray[Any],
    class_names: Optional[List[str]] = None
) -> None:
    """
    Analiza y loggea la distribución de clases en train/test.
    
    Útil para detectar desbalances que puedan causar problemas.
    
    Args:
        y_train: Labels de entrenamiento
        y_test: Labels de prueba
        class_names: Nombres de clases (opcional)
    """
    unique_train, counts_train = np.unique(y_train, return_counts=True)
    unique_test, counts_test = np.unique(y_test, return_counts=True)
    
    logger.info("=" * 60)
    logger.info("DISTRIBUCIÓN DE CLASES")
    logger.info("=" * 60)
    
    for i, class_label in enumerate(unique_train):
        name = class_names[i] if class_names else f"Clase {class_label}"
        train_count = counts_train[i]
        test_count = counts_test[np.where(unique_test == class_label)[0][0]] \
            if class_label in unique_test else 0
        
        train_pct = (train_count / len(y_train)) * 100
        test_pct = (test_count / len(y_test)) * 100 if test_count > 0 else 0
        
        logger.info(
            f"{name:20s} | Train: {train_count:5d} ({train_pct:5.2f}%) | "
            f"Test: {test_count:5d} ({test_pct:5.2f}%)"
        )
    
    logger.info("=" * 60)


def calculate_split_statistics(
    X_train: npt.NDArray[Any],
    X_test: npt.NDArray[Any]
) -> dict[str, float]:
    """
    Calcula estadísticas del split para validación.
    
    Returns:
        Dict con métricas: train_ratio, test_ratio, feature_variance_ratio
    
    Decisión: Retornar dict en vez de múltiples valores para extensibilidad.
    """
    total_samples = len(X_train) + len(X_test)
    
    stats = {
        'train_ratio': len(X_train) / total_samples,
        'test_ratio': len(X_test) / total_samples,
        'train_samples': len(X_train),
        'test_samples': len(X_test),
        'total_samples': total_samples
    }
    
    # Calcular ratio de varianzas (detectar si el split está sesgado)
    if X_train.ndim > 1:
        train_var = np.var(X_train, axis=0).mean()
        test_var = np.var(X_test, axis=0).mean()
        stats['variance_ratio'] = test_var / train_var if train_var > 0 else 0.0
    
    return stats

Por qué estas funciones:

  • analyze_split_distribution: En clasificación desbalanceada, esto te salva. Si tenés 1000 muestras de clase A y 10 de clase B, y el split te deja 0 de clase B en test... estás frito.
  • calculate_split_statistics: El variance_ratio te dice si tu split está sesgado. Si es muy diferente de 1.0, algo raro pasó.

Paso 5: Ejemplo de Uso Real

def main_example() -> None:
    """
    Ejemplo completo de uso con diferentes escenarios.
    """
    # Generar datos sintéticos
    np.random.seed(42)
    n_samples = 1000
    n_features = 20
    
    X = np.random.randn(n_samples, n_features)
    y = np.random.randint(0, 3, size=n_samples)  # 3 clases
    
    print("\n" + "="*70)
    print("EJEMPLO 1: Split básico sin estratificación")
    print("="*70)
    
    config_basic = SplitConfig(
        test_size=0.2,
        random_state=42,
        shuffle=True,
        stratify_strategy=StratifyStrategy.NONE
    )
    
    splitter_basic = TrainTestSplitter(config=config_basic)
    X_train, X_test, y_train, y_test = splitter_basic.split(X, y)
    
    stats = calculate_split_statistics(X_train, X_test)
    print(f"\nEstadísticas del split:")
    print(f"  Train ratio: {stats['train_ratio']:.3f}")
    print(f"  Test ratio: {stats['test_ratio']:.3f}")
    print(f"  Variance ratio: {stats.get('variance_ratio', 'N/A'):.3f}")
    
    if y_train is not None and y_test is not None:
        analyze_split_distribution(y_train, y_test)
    
    print("\n" + "="*70)
    print("EJEMPLO 2: Split con estratificación automática")
    print("="*70)
    
    config_stratified = SplitConfig(
        test_size=0.25,
        random_state=123,
        shuffle=True,
        stratify_strategy=StratifyStrategy.AUTO
    )
    
    splitter_stratified = TrainTestSplitter(config=config_stratified)
    X_train_s, X_test_s, y_train_s, y_test_s = splitter_stratified.split(X, y)
    
    if y_train_s is not None and y_test_s is not None:
        analyze_split_distribution(y_train_s, y_test_s)
    
    print("\n" + "="*70)
    print("EJEMPLO 3: Manejo de error - Datos insuficientes")
    print("="*70)
    
    try:
        X_tiny = np.random.randn(5, 10)
        config_error = SplitConfig(test_size=0.8)  # 80% test con solo 5 muestras
        splitter_error = TrainTestSplitter(config=config_error)
        splitter_error.split(X_tiny)
    except InsufficientDataError as e:
        print(f"❌ Error capturado correctamente: {e}")


if __name__ == "__main__":
    main_example()

Casos Edge que Te Van a Dar Dolores de Cabeza

1. Datasets Desbalanceados

El problema:

# Dataset con 950 de clase 0, 50 de clase 1
y = np.concatenate([np.zeros(950), np.ones(50)])

Sin estratificación, podés terminar con 0 muestras de clase 1 en tu test set. Tu modelo nunca va a ver esa clase en evaluación.

La solución:

config = SplitConfig(
    test_size=0.2,
    stratify_strategy=StratifyStrategy.AUTO  # Mantiene proporciones
)

2. Series Temporales

ERROR FATAL:

# NUNCA hagas esto con datos temporales
X_train, X_test = train_test_split(time_series_data, shuffle=True)

Por qué: Estás entrenando con datos del futuro y evaluando con datos del pasado. En producción, solo tenés acceso al pasado para predecir el futuro.

La solución:

# Para series temporales, split manual por fecha
split_idx = int(len(data) * 0.8)
train_data = data[:split_idx]  # 80% más antiguo
test_data = data[split_idx:]   # 20% más reciente

3. Data Leakage en Features Derivadas

El error silencioso:

# MAL - Normalizar antes de split
from sklearn.preprocessing import StandardScaler

scaler = StandardScaler()
X_normalized = scaler.fit_transform(X)  # ❌ Usa estadísticas de TODO el dataset
X_train, X_test = train_test_split(X_normalized)

Por qué está mal: El scaler vió las estadísticas (media, std) del test set. En producción, no vas a tener esa información.

Bien hecho:

# BIEN - Split primero, normalizar después
X_train, X_test = train_test_split(X)

scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)  # ✅ Solo aprende de train
X_test_scaled = scaler.transform(X_test)         # ✅ Aplica lo aprendido

La Matemática del Tamaño del Test Set

¿Por qué 80/20 es tan común? No es arbitrario, hay matemática detrás.

Trade-off Sesgo-Varianza en la Estimación

El error de estimación del test set tiene dos componentes:

$$\text{Error}_{\text{estimación}} = \text{Bias}^2 + \text{Variance}$$

**Si el test set es muy pequeño (**$\alpha$ muy chico):

  • Varianza alta: Pocas muestras → estimación inestable
  • Sesgo bajo: Más datos para entrenar → mejor modelo

**Si el test set es muy grande (**$\alpha $ muy grande):

  • Varianza baja: Muchas muestras → estimación estable
  • Sesgo alto: Pocos datos para entrenar → modelo subóptimo

Regla de Oro Empírica

Para un dataset de tamaño $N$:

$$|\mathcal{D}_{\text{test}}| \geq \frac{30}{n_{\text{classes}}}$$

Ejemplo: 3 clases → necesitás mínimo 10 muestras por clase en test → 30 muestras totales.

Proporción óptima según tamaño:

  • $N < 500$: Considerá usar k-fold cross-validation en vez de simple split
  • $500 \leq N < 5000$: $\alpha = 0.2$ (80/20)
  • $5000 \leq N < 50000$: $\alpha = 0.15$ (85/15)
  • $N \geq 50000$: $\alpha = 0.1$ (90/10)

Razón: Con más datos, cada muestra aporta menos información marginal. Podés permitirte un test set proporcionalmente más chico.

Cómo Evitar Que Te Embalen

Checklist antes de hacer el split:

  1. ✅ ¿Tus datos son temporales? → No uses shuffle
  2. ✅ ¿Tenés clases desbalanceadas? → Usa stratify
  3. ✅ ¿Vas a normalizar/escalar? → Split primero, scale después
  4. ✅ ¿Tenés pocas muestras? → Considerá k-fold CV
  5. ✅ ¿Fijaste random_state? → Reproducibilidad en debugging
  6. ✅ ¿Loggeaste la distribución? → Detectá problemas temprano

Conclusión: De Una, Parcero

El train/test split es la base de todo. Si esto lo hacés mal, podés tener un modelo que en desarrollo muestre 99% accuracy y en producción no le pegue ni a una.

La implementación que te mostré no es solo "código bonito". Cada validación, cada excepción custom, cada log, está ahí porque ya me pasó que algo se cayó por no tenerlo.

Recordá:

  • Evaluá con datos que el modelo nunca vió
  • Estratificá cuando tengas desbalance
  • No hagas shuffle en series temporales (por Dios)
  • Split antes de cualquier preprocesamiento que vea estadísticas

Hágale con confianza. Esta vaina es crítica, pero con la implementación correcta, es una de las cosas más sólidas de tu pipeline.
