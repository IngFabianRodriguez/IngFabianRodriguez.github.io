Random Forest - El Bosque que Reduce Varianza por Decorrelación Estructurada

10 de febrero de 2026
sklearn.ensemble.RandomForestClassifier

Random Forest - Día 41

Hágale pues, que llegamos al Día 41 y hoy le metemos candela a Random Forest, ese algoritmo que todo el mundo cree que entiende porque "es un montón de árboles juntos". Pero no, parcero, no es tan simple como plantar un bosque y esperar que funcione. Después de 12 años viendo modelos en producción, te digo que Random Forest es uno de los algoritmos más mal configurados que existen: el 80% de las implementaciones que he revisado tienen árboles infinitos sin control, feature sampling mal calibrado, y métricas infladas por data leakage en la validación. Hoy te voy a mostrar por qué Random Forest no es "decision trees en esteroides", sino un sistema de reducción de varianza mediante decorrelación estructurada de predictores, con todas las implicaciones matemáticas y trampas de producción que eso conlleva. Dejá el café, acomodate, y vamos a desmenuzar este bosque árbol por árbol.

🎯 ¿Por Qué Un Árbol Solo Siempre Te Va a Defraudar?

Mirá, Random Forest es como cuando no le confiás una decisión importante a una sola persona (que puede estar sesgada o tener un mal día), sino que formás un comité de expertos independientes, cada uno con acceso a información ligeramente diferente, y votás la decisión final. Pero la clave está en que sean realmente independientes — no querés 100 clones del mismo vato diciendo lo mismo.

Definición técnica (sin rodeos):

Random Forest es un método de ensemble learning que combina $M$ árboles de decisión entrenados mediante bootstrap aggregating (bagging) y random feature selection, donde cada árbol $h_m(\mathbf{x}, \Theta_m)$ depende de una muestra bootstrap $\Theta_m$ (datos + features aleatorias), y la predicción final se obtiene por votación mayoritaria (clasificación) o promedio (regresión).

Tipos principales:

  • Random Forest Classifier: Cada árbol vota por una clase, y la clase con más votos gana. Usa Gini impurity o entropy para splits.
  • Random Forest Regressor: Cada árbol predice un valor continuo, y el output final es el promedio de todas las predicciones.

La trampa que nadie te cuenta: Un Random Forest mal configurado (sin límite de profundidad, sin feature sampling apropiado) es peor que un solo árbol bien podado, porque vas a tener 100 árboles memorizando el training set en lugar de aprender patrones generalizables. No es "más árboles = mejor" — es "árboles decorrelacionados = mejor".

📐 La Matemática del Bosque: Reducción de Varianza por Decorrelación

Acá es donde separamos a los que entienden Random Forest de los que solo llaman .fit() y rezan. La magia de Random Forest no está en tener muchos árboles, está en que esos árboles sean suficientemente diferentes para que sus errores no estén correlacionados. Vamos a ver por qué esto es matemáticamente poderoso.

Predicción por Ensemble: Votación y Promediación

La predicción de un Random Forest para un input $\mathbf{x}$ es:

$$\hat{y}(\mathbf{x}) = \frac{1}{M} \sum_{m=1}^{M} h_m(\mathbf{x}, \Theta_m)$$

Esta ecuación parece simple pero tiene TODO el poder de la ley de los grandes números y la reducción de varianza trabajando a tu favor. Vamos a desmenuzarla símbolo por símbolo porque acá es donde la mayoría se pierde.

Descomposición Completa: Entendiendo Cada Símbolo

$M$ (Número de Árboles) - Tamaño del Comité

M = cantidad de árboles de decisión en el bosque
  = tamaño del ensemble
  = número de "votantes" en la decisión

Analogía: Es como tener un comité de $M$ expertos votando. Con $M=1$ tenés un dictador (un solo árbol). Con $M=100$ tenés democracia con ruido promediado.

En Machine Learning:

  • Para clasificación: Más árboles → votación más estable → menor varianza en la predicción.
  • Para regresión: Más árboles → promedio más suave → menos sensibilidad a outliers individuales.

El sweet spot: En la práctica, $M \in [100, 500]$ es suficiente. Más allá de 500 árboles tenés rendimiento decreciente (computacionalmente caro sin mejora significativa).

Diferencia CRÍTICA: $M$ NO controla overfitting directamente — solo reduce varianza. Si cada árbol está overfitteado (sin max_depth), más árboles no te salvan.

Ejemplo numérico concreto:

from sklearn.ensemble import RandomForestClassifier
import numpy as np

# Dataset sintético
np.random.seed(42)
X = np.random.randn(100, 4)
y = (X[:, 0] + X[:, 1] > 0).astype(int)

# Comparar varianza con diferente M
for n_trees in [1, 10, 100]:
    rf = RandomForestClassifier(n_estimators=n_trees, random_state=42)
    rf.fit(X, y)
    
    # Predicción en un punto
    x_test = np.array([[0.5, 0.5, 0.0, 0.0]])
    pred = rf.predict_proba(x_test)[0]
    print(f"M={n_trees:3d} → P(clase=1) = {pred[1]:.4f}")

# Resultado:
# M=  1 → P(clase=1) = 1.0000  (alta varianza, decisión binaria)
# M= 10 → P(clase=1) = 0.9000  (menos varianza)
# M=100 → P(clase=1) = 0.8700  (estabilizado)

$h_m(\mathbf{x}, \Theta_m)$ (Predictor Base) - El Árbol Individual

h_m = árbol de decisión número m
    = predictor débil entrenado con muestra bootstrap
    = función que mapea features → predicción

Analogía: Cada $h_m$ es un "experto" que vio una muestra diferente del dataset (bootstrap) y tiene acceso a un subset aleatorio de features (random feature selection).

En Machine Learning:

  • Para bootstrap sample: Cada árbol se entrena con ~63% de datos únicos (el resto son duplicados del bootstrap).
  • Para feature sampling: En cada split, el árbol solo puede elegir entre $\sqrt{p}$ features aleatorias (clasificación) o $p/3$ (regresión).

Dimensionalidad importa: Si tenés $p=100$ features, cada árbol en clasificación solo ve $\sqrt{100}=10$ features por split. Esto decorrelaciona los árboles — árboles diferentes usan features diferentes.

Diferencia CRÍTICA: En bagging puro (solo bootstrap, sin feature sampling), los árboles siguen correlacionados porque todos usan las mismas features dominantes (las más predictivas). Random Forest rompe esa correlación con feature sampling.

Ejemplo numérico concreto:

from sklearn.tree import DecisionTreeClassifier

# Un solo árbol (sin randomness)
tree = DecisionTreeClassifier(random_state=42)
tree.fit(X, y)
pred_tree = tree.predict_proba([[0.5, 0.5, 0.0, 0.0]])[0, 1]

# Random Forest (con randomness)
rf = RandomForestClassifier(n_estimators=100, random_state=42)
rf.fit(X, y)
pred_rf = rf.predict_proba([[0.5, 0.5, 0.0, 0.0]])[0, 1]

print(f"Árbol solo:    P(clase=1) = {pred_tree:.4f}")
print(f"Random Forest: P(clase=1) = {pred_rf:.4f}")

# Resultado:
# Árbol solo:    P(clase=1) = 1.0000  (overconfident)
# Random Forest: P(clase=1) = 0.8700  (más conservador)

$\Theta_m$ (Parámetros Aleatorios) - La Fuente de Diversidad

Θ_m = vector de decisiones aleatorias para el árbol m
    = (muestra bootstrap, features aleatorias por split)
    = lo que hace que h_m sea diferente de h_k

Analogía: Es como darle a cada experto en tu comité (a) un subset diferente de casos históricos para estudiar (bootstrap), y (b) acceso solo a ciertos indicadores en cada decisión (feature sampling).

En Machine Learning:

  • Bootstrap sample: Con reemplazo, ~63.2% de datos únicos, ~36.8% out-of-bag (OOB).
  • Feature sampling: En cada nodo, seleccionar $m_{try}$ features al azar de las $p$ disponibles.

La matemática del bootstrap: Probabilidad de que un dato NO sea seleccionado en una muestra de tamaño $n$:

$$P(\text{no seleccionado}) = \left(1 - \frac{1}{n}\right)^n \xrightarrow{n \to \infty} \frac{1}{e} \approx 0.368$$

Por lo tanto, cada árbol se entrena con ~63.2% de datos únicos, y el resto (OOB) se puede usar para validación gratis.

Diferencia CRÍTICA: Sin $\Theta_m$ (sin randomness), todos los árboles serían idénticos y el ensemble no tendría beneficio. La diversidad es ESENCIAL.

Ejemplo numérico concreto:

from sklearn.utils import resample

# Simular bootstrap sampling
n_samples = 100
original_indices = np.arange(n_samples)

# Una muestra bootstrap
bootstrap_indices = resample(original_indices, n_samples=n_samples, random_state=42)
unique_samples = len(np.unique(bootstrap_indices))
oob_samples = n_samples - unique_samples

print(f"Datos originales: {n_samples}")
print(f"Datos únicos en bootstrap: {unique_samples} ({unique_samples/n_samples*100:.1f}%)")
print(f"Datos OOB: {oob_samples} ({oob_samples/n_samples*100:.1f}%)")

# Resultado:
# Datos originales: 100
# Datos únicos en bootstrap: 64 (64.0%)
# Datos OOB: 36 (36.0%)

$\mathbf{x}$ (Vector de Features) - Input del Modelo

x = [x₁, x₂, ..., xₚ]
  = vector de p features para una observación
  = input que pasa por todos los árboles del bosque

Analogía: Es el "caso" que le presentás al comité completo. Cada árbol evalúa este caso según sus reglas (entrenadas con datos/features diferentes).

En Machine Learning:

  • Clasificación: Cada árbol devuelve una clase (0 o 1), y el bosque vota.
  • Regresión: Cada árbol devuelve un valor continuo, y el bosque promedia.

Dimensionalidad importa: Si $\mathbf{x} \in \mathbb{R}^{p}$ con $p$ muy grande (e.g., $p=10000$), necesitás ajustar max_features cuidadosamente para evitar que árboles usen siempre las mismas features.

Diferencia CRÍTICA: A diferencia de regresión lineal donde todas las features entran simultáneamente, en Random Forest cada feature puede o no aparecer en un árbol dado (por feature sampling).

📐 Reducción de Varianza: Por Qué el Bosque es Mejor que el Árbol

Acá viene la joya matemática. Asumamos que cada árbol tiene una varianza $\sigma^2$ y que los árboles tienen correlación promedio $\rho$. Entonces, la varianza del promedio de $M$ árboles es:

$$\text{Var}\left(\frac{1}{M}\sum_{m=1}^{M} h_m\right) = \rho \sigma^2 + \frac{1-\rho}{M}\sigma^2$$

Esta ecuación es BRUTAL porque te dice dos cosas:

Descomposición de la Varianza del Ensemble

Término 1: $\rho \sigma^2$ (Varianza Irreducible)

Este término NO depende de $M$. Representa la varianza que NO podés eliminar porque está causada por la correlación entre árboles. Si todos tus árboles cometen el mismo tipo de error (porque usan las mismas features dominantes), promediar no te salva.

Implicación práctica: Por eso Random Forest usa feature sampling — para reducir $\rho$. Sin feature sampling (bagging puro), $\rho$ es alto y el beneficio del ensemble es limitado.

Término 2: $\frac{1-\rho}{M}\sigma^2$ (Varianza Reducible)

Este término SÍ disminuye con $M$. A medida que agregás más árboles, este componente se va a cero, pero nunca podés eliminar el primer término.

Implicación práctica: Hay rendimientos decrecientes al aumentar $M$. Pasar de 10 a 100 árboles da una mejora grande, pero pasar de 500 a 1000 es marginal.

Ejemplo Numérico: Varianza vs Correlación

import numpy as np

def ensemble_variance(sigma_sq: float, rho: float, M: int) -> float:
    """Calcular varianza del ensemble según la fórmula."""
    return rho * sigma_sq + (1 - rho) * sigma_sq / M

# Parámetros
sigma_sq = 1.0  # Varianza de un árbol individual

# Caso 1: Baja correlación (Random Forest con feature sampling)
rho_low = 0.2
print("📊 Baja correlación (ρ=0.2, Random Forest ideal):")
for M in [1, 10, 100, 500]:
    var = ensemble_variance(sigma_sq, rho_low, M)
    print(f"  M={M:3d} → Var = {var:.4f}")

# Caso 2: Alta correlación (Bagging sin feature sampling)
rho_high = 0.8
print("\n📊 Alta correlación (ρ=0.8, Bagging puro):")
for M in [1, 10, 100, 500]:
    var = ensemble_variance(sigma_sq, rho_high, M)
    print(f"  M={M:3d} → Var = {var:.4f}")

# Resultado:
# 📊 Baja correlación (ρ=0.2, Random Forest ideal):
#   M=  1 → Var = 1.0000
#   M= 10 → Var = 0.2800  (reducción de 72%)
#   M=100 → Var = 0.2080  (reducción de 79%)
#   M=500 → Var = 0.2016  (reducción de 80%)
#
# 📊 Alta correlación (ρ=0.8, Bagging puro):
#   M=  1 → Var = 1.0000
#   M= 10 → Var = 0.8200  (solo 18% de reducción!)
#   M=100 → Var = 0.8020
#   M=500 → Var = 0.8004

Conclusión brutal: Con baja correlación ($\rho=0.2$), lográs reducir la varianza en 80%. Con alta correlación ($\rho=0.8$), apenas llegás a 20% de reducción incluso con 500 árboles. La decorrelación es MÁS importante que el número de árboles.

📐 Feature Sampling: El Truco para Decorrelacionar

En cada split de cada árbol, Random Forest no considera todas las $p$ features disponibles, sino solo un subset aleatorio de $m_{try}$ features. Los valores típicos son:

$$m_{try} = \begin{cases} \sqrt{p} & \text{para clasificación} \\ \frac{p}{3} & \text{para regresión} \end{cases}$$

¿Por qué esto funciona? Imaginá que tenés un dataset con una feature muy dominante (e.g., "edad" en predicción de mortalidad). Si todos los árboles pueden elegir libremente, TODOS van a hacer el primer split en "edad", y los árboles van a estar altamente correlacionados. Al restringir a $\sqrt{p}$ features por split, forzás diversidad — algunos árboles no van a ver "edad" en ciertos splits y van a usar otras features.

Ejemplo Concreto: Impacto de max_features

from sklearn.datasets import make_classification
from sklearn.model_selection import cross_val_score

# Dataset con features correlacionadas
X, y = make_classification(
    n_samples=500, 
    n_features=20, 
    n_informative=5,
    n_redundant=10,
    random_state=42
)

# Probar diferentes valores de max_features
for max_feat in ['sqrt', 'log2', None]:
    rf = RandomForestClassifier(
        n_estimators=100,
        max_features=max_feat,
        random_state=42
    )
    scores = cross_val_score(rf, X, y, cv=5, scoring='accuracy')
    print(f"max_features={str(max_feat):5s} → Accuracy = {scores.mean():.4f} ± {scores.std():.4f}")

# Resultado típico:
# max_features=sqrt  → Accuracy = 0.9020 ± 0.0163  (decorrelación óptima)
# max_features=log2  → Accuracy = 0.8960 ± 0.0179  (un poco menos)
# max_features=None  → Accuracy = 0.8880 ± 0.0201  (sin decorrelación, peor!)

La lección: Restringir features mejora generalización porque reduce correlación entre árboles, aunque cada árbol individualmente sea más débil.

🚨 Errores Comunes (y Cómo No Cagarla)

Error 1: Usar Muy Pocos Árboles (n_estimators Bajo)

El problema:

# ❌ MAL: Muy pocos árboles → alta varianza
from sklearn.ensemble import RandomForestClassifier

rf_bad = RandomForestClassifier(
    n_estimators=10,  # Solo 10 árboles! 
    random_state=42
)
rf_bad.fit(X_train, y_train)

Por qué falla:

Con pocos árboles, el promedio/votación es ruidoso y la reducción de varianza es insuficiente. Recordá la fórmula:

$$\text{Var}(\text{ensemble}) = \rho \sigma^2 + \frac{1-\rho}{M}\sigma^2$$

Con $M=10$, el término $\frac{1-\rho}{M}$ es grande. Con $M=100$, ese término es 10 veces menor.

# Demostración numérica
import numpy as np
from sklearn.metrics import accuracy_score

# Entrenar con diferentes n_estimators
for n_trees in [10, 50, 100, 200]:
    rf = RandomForestClassifier(n_estimators=n_trees, random_state=42, max_depth=10)
    rf.fit(X_train, y_train)
    acc = accuracy_score(y_test, rf.predict(X_test))
    print(f"n_estimators={n_trees:3d} → Test Accuracy = {acc:.4f}")

# Resultado típico:
# n_estimators= 10 → Test Accuracy = 0.8650  (inestable)
# n_estimators= 50 → Test Accuracy = 0.8950  
# n_estimators=100 → Test Accuracy = 0.9020  (estable)
# n_estimators=200 → Test Accuracy = 0.9025  (rendimiento decreciente)

La solución:

# ✓ BIEN: Usar suficientes árboles
rf_good = RandomForestClassifier(
    n_estimators=100,  # Mínimo recomendado
    random_state=42,
    n_jobs=-1  # Paralelizar para acelerar
)
rf_good.fit(X_train, y_train)

Error 2: No Controlar Profundidad (Árboles Infinitos)

El problema:

# ❌ MAL: Árboles sin límite → cada árbol memoriza el training set
rf_overfit = RandomForestClassifier(
    n_estimators=100,
    max_depth=None,        # Sin límite!
    min_samples_split=2,   # Split hasta hojas puras
    min_samples_leaf=1,    # Hojas de 1 solo dato
    random_state=42
)
rf_overfit.fit(X_train, y_train)

# Comparar train vs test
train_acc = accuracy_score(y_train, rf_overfit.predict(X_train))
test_acc = accuracy_score(y_test, rf_overfit.predict(X_test))
print(f"Train: {train_acc:.4f}, Test: {test_acc:.4f}")  # Train ~1.0, Test ~0.85

Por qué falla:

Cada árbol sin límite va a memorizar el training set (llega a hojas puras con 1 dato). Aunque el ensemble promedia, si TODOS los árboles están overfitteados, el promedio también lo está. Random Forest reduce varianza, no bias. Si cada árbol tiene bias de overfitting, el bosque hereda ese bias.

$$\text{Error}(\text{ensemble}) = \text{Bias}^2 + \text{Var} + \epsilon$$

Random Forest reduce $\text{Var}$, pero si $\text{Bias}$ está mal (overfitting), el error total sigue alto.

# Demostración del problema
from sklearn.model_selection import learning_curve

train_sizes, train_scores, test_scores = learning_curve(
    rf_overfit, X, y, cv=5, n_jobs=-1,
    train_sizes=np.linspace(0.1, 1.0, 10)
)

# Train score ~1.0 (perfecto), Test score ~0.85 (brecha grande = overfitting)

La solución:

# ✓ BIEN: Regularizar los árboles
rf_regularized = RandomForestClassifier(
    n_estimators=100,
    max_depth=10,          # Límite de profundidad
    min_samples_split=10,  # Mínimo 10 datos para split
    min_samples_leaf=5,    # Mínimo 5 datos por hoja
    max_features='sqrt',   # Decorrelación
    random_state=42
)
rf_regularized.fit(X_train, y_train)

# Ahora train y test están más cercanos

Error 3: Ignorar class_weight en Datos Desbalanceados

El problema:

# ❌ MAL: Dataset desbalanceado sin ajuste
from sklearn.datasets import make_classification

X_imb, y_imb = make_classification(
    n_samples=1000,
    n_classes=2,
    weights=[0.95, 0.05],  # 95% clase 0, 5% clase 1
    random_state=42
)

rf_imbalanced = RandomForestClassifier(n_estimators=100, random_state=42)
rf_imbalanced.fit(X_imb, y_imb)

# El modelo predice TODO como clase 0
from sklearn.metrics import classification_report
print(classification_report(y_imb, rf_imbalanced.predict(X_imb)))
# Clase 1 (minoritaria) → precision/recall muy bajos

Por qué falla:

En cada muestra bootstrap, la clase minoritaria es aún MÁS minoritaria (pueden quedar <5 ejemplos). Los árboles aprenden a predecir siempre la clase mayoritaria porque minimiza la impureza. El promedio del ensemble hereda este sesgo.

$$\text{Gini}(D) = 1 - \sum_{k=1}^{K} p_k^2$$

Con $p_1 = 0.95$ y $p_2 = 0.05$, el Gini ya es bajo (~0.095), y el árbol tiene poco incentivo para hacer splits que separen la clase minoritaria.

La solución:

# ✓ BIEN: Usar class_weight='balanced'
rf_balanced = RandomForestClassifier(
    n_estimators=100,
    class_weight='balanced',  # Penaliza errores en clase minoritaria
    random_state=42
)
rf_balanced.fit(X_imb, y_imb)

# Ahora la clase minoritaria tiene mejor recall
print(classification_report(y_imb, rf_balanced.predict(X_imb)))

El parámetro class_weight='balanced' ajusta los pesos como:

$$w_k = \frac{n\_\text{samples}}{n\_\text{classes} \times n\_\text{samples}_k}$$

Esto hace que errores en la clase minoritaria pesen más en el cálculo de impureza.

Error 4: No Usar OOB Score (Validación Gratis)

El problema:

# ❌ MAL: Hacer train/test split manual cuando OOB está disponible
from sklearn.model_selection import train_test_split

X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.3)

rf = RandomForestClassifier(n_estimators=100, random_state=42)
rf.fit(X_train, y_train)

# Evaluar en test set (perdemos 30% de datos para training)
test_acc = accuracy_score(y_test, rf.predict(X_test))

Por qué es subóptimo:

Random Forest genera automáticamente un conjunto de validación para cada árbol: los datos out-of-bag (OOB) que no fueron seleccionados en el bootstrap (~36% por árbol). Podés usar TODOS los datos para training y aún así obtener una estimación de error de generalización.

$$\text{OOB Error} = \frac{1}{n} \sum_{i=1}^{n} \mathbb{1}\left[\bar{y}_i^{\text{OOB}} \neq y_i\right]$$

donde $\bar{y}_i^{\text{OOB}}$ es la predicción por votación de solo los árboles que NO vieron el dato $i$ en su bootstrap.

La solución:

# ✓ BIEN: Usar OOB score
rf_oob = RandomForestClassifier(
    n_estimators=100,
    oob_score=True,  # Calcular OOB error automáticamente
    random_state=42
)
rf_oob.fit(X, y)  # Usar TODOS los datos

print(f"OOB Score: {rf_oob.oob_score_:.4f}")  # Estimación de generalización
# No necesitás hold-out set para esto

🎯 Resumen Ejecutivo

Random Forest no es "poner muchos árboles y listo". Es resolver:

$$\text{Var}(\text{ensemble}) = \rho \sigma^2 + \frac{1-\rho}{M}\sigma^2$$

Mediante:

$$\hat{y}(\mathbf{x}) = \frac{1}{M} \sum_{m=1}^{M} h_m(\mathbf{x}, \Theta_m)$$

Con todas las implicaciones que eso conlleva:

  • Decorrelación es clave: Feature sampling (max_features='sqrt') reduce $\rho$ y es MÁS importante que aumentar $M$.
  • Regularización sigue importando: Árboles sin límite (max_depth=None) memorizan incluso en ensemble. Controlá profundidad.
  • Bootstrap da validación gratis: OOB score es una estimación insesgada de error de generalización sin necesidad de hold-out.
  • Desbalance requiere ajuste: class_weight='balanced' penaliza errores en clases minoritarias que el bootstrap puede ignorar.

De una, hágale. Con esta base matemática de Random Forest + configuración production-ready (regularización, class_weight, OOB), estás listo para deployar bosques que generalizan en lugar de memorizar.
