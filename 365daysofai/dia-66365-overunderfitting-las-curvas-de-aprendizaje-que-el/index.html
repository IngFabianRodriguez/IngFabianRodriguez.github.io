Over/Underfitting - Día 66

Hágale pues, parcero. Llegamos al Día 66 del reto y hoy toca hablar de algo que técnicamente "todos saben" pero que el 85% de los equipos con los que he trabajado en mis 12 años de carrera siguen cagando en producción. Over/Underfitting no es "ese concepto introductorio que ya superé". Es el diagnóstico que separa a un ingeniero que entrena modelos de uno que entiende por qué sus modelos fallan. He visto dashboards de producción con accuracy del 99% en entrenamiento y 61% en producción. He visto reentrenamientos semanales que no resuelven nada porque nadie se tomó 15 minutos a graficar una curva de aprendizaje. Dejá el café, acomodate, y vamos a entender esto de verdad.

🎯 ¿Tu modelo aprende o memoriza?

Mirá, el overfitting es como ese parcero que estudia únicamente los exámenes anteriores de palabra por palabra. El día del examen con preguntas nuevas: cero. Ha memorizado respuestas, no conceptos. El underfitting, en cambio, es el que ni siquiera leyó el libro — llega al examen sin la base mínima para responder nada bien.

Definición técnica (sin rodeos):

Dado un modelo $f_\theta$ entrenado sobre un conjunto $\mathcal{D}_{train}$ de $n$ ejemplos, buscamos que el error de generalización $\mathcal{E}_{gen}$ sea mínimo, donde:

$$\mathcal{E}_{gen} = \mathbb{E}_{(x,y) \sim P_{data}}\left[\mathcal{L}(f_\theta(x), y)\right]$$

Los dos fracasos:

  • Underfitting (sesgo alto): El modelo tiene $\mathcal{E}_{train}$ alto y $\mathcal{E}_{val}$ alto. La hipótesis es demasiado simple para capturar el patrón real. Capacidad insuficiente.
  • Overfitting (varianza alta): El modelo tiene $\mathcal{E}_{train}$ bajo pero $\mathcal{E}_{val}$ alto. Memoriza ruido del training set. Capacidad excesiva sin regularización.

La trampa es creer que el problema siempre es uno de los dos. En producción, frecuentemente coexisten dependiendo del subconjunto de datos que analices. Las curvas de aprendizaje son la herramienta que te permite ver cuál es el problema real antes de tomar una decisión.

📐 El Trade-off Bias-Varianza

Esta es la ecuación que ningún tutorial te explica completo porque requiere que te sientes a pensar. El error esperado de un modelo sobre un punto $x$ se puede descomponer así:

$$\mathbb{E}\left[\mathcal{L}(f_\theta(x), y)\right] = \underbrace{\text{Bias}^2[f_\theta(x)]}_{\text{sesgo}} + \underbrace{\text{Var}[f_\theta(x)]}_{\text{varianza}} + \underbrace{\sigma^2_\epsilon}_{\text{ruido irreducible}}$$

Esta ecuación parece simple pero tiene TODO el poder diagnóstico que necesitás. Vamos a desmenuzarla símbolo por símbolo porque acá es donde la mayoría se pierde.

Descomposición Completa: Entendiendo Cada Símbolo

$\text{Bias}^2[f_\theta(x)]$ (Sesgo al Cuadrado) — Propósito Simple

Bias² = (E[f_θ(x)] - y_real)²
      = distancia entre la predicción promedio y la verdad

Analogía: Es como una báscula descalibrada que siempre te da 3 kg más de lo real. No importa cuántas veces te pesés — el error sistemático está ahí.

En Machine Learning:

  • Bias alto: Modelo demasiado simple (regresión lineal sobre datos cuadráticos). Subestima la complejidad real.
  • Bias bajo: El modelo tiene suficiente capacidad para aproximar la función real.

Diferencia CRÍTICA: El bias no cambia al agregar más datos de entrenamiento. Es un problema de arquitectura, no de cantidad.

Ejemplo numérico concreto:

import numpy as np

np.random.seed(42)
x = np.linspace(0, 3, 100)
y_real = np.sin(x * np.pi)  # función real no lineal

# Modelo con bias alto: intenta aproximar sin(πx) con una línea recta
coef = np.polyfit(x, y_real, deg=1)  # grado 1 → lineal
y_pred_lineal = np.polyval(coef, x)

bias_squared = (np.mean(y_pred_lineal) - np.mean(y_real)) ** 2
print(f"Bias²: {bias_squared:.6f}")  # alto, sistemático

$\text{Var}[f_\theta(x)]$ (Varianza) — Propósito Simple

Var  = E[(f_θ(x) - E[f_θ(x)])²]
     = cuánto cambia la predicción si cambian los datos de entrenamiento

Analogía: Un modelo con varianza alta es como un GPS que te da rutas completamente diferentes cada vez que recalcula aunque estés en el mismo punto — depende demasiado del tráfico en tiempo real (ruido).

En Machine Learning:

  • Varianza alta: Modelo complejo (árbol de decisión sin poda, KNN con k=1). Overfitting garantizado.
  • Varianza baja: El modelo es estable frente a perturbaciones en los datos de entrada.

Diferencia CRÍTICA: La varianza sí puede reducirse con más datos. El bias no.

$\sigma^2_\epsilon$ (Ruido Irreducible) — Propósito Simple

σ²_ε = Var[y | x]
     = varianza inherente en los datos que ningún modelo puede eliminar

Analogía: Es como querer predecir el resultado de un dado perfecto. No hay modelo que lo logre porque el proceso es inherentemente aleatorio.

Este término define el piso mínimo de error que cualquier modelo puede alcanzar. Si tu curva de validación nunca baja de ese piso, no es problema tuyo — es el problema.

📐 Las Curvas de Aprendizaje como Diagnóstico

La curva de aprendizaje grafica $\mathcal{E}_{train}(n)$ y $\mathcal{E}_{val}(n)$ en función del tamaño del conjunto de entrenamiento $n$:

$$\mathcal{E}(n) = \frac{1}{n} \sum_{i=1}^{n} \mathcal{L}(f_\theta(x_i), y_i)$$

Diagnóstico por forma de curva:

Síntoma Causa Señal visual
$\mathcal{E}_{train} \approx \mathcal{E}_{val}$ altos Underfitting (bias alto) Dos curvas juntas y altas
$\mathcal{E}_{train} \ll \mathcal{E}_{val}$ Overfitting (varianza alta) Brecha grande entre curvas
$\mathcal{E}_{val}$ decrece con $n$ Varianza alta Más datos ayudan
$\mathcal{E}_{val}$ estancada con $n$ Bias alto Más datos no ayudan

Acá viene la implementación production-ready. Esto es lo que realmente deberías tener en tu toolkit, no un plt.plot(history) improvisado.

from __future__ import annotations

import logging
from dataclasses import dataclass, field
from typing import Literal

import matplotlib.pyplot as plt
import matplotlib.ticker as mticker
import numpy as np
from sklearn.base import BaseEstimator
from sklearn.model_selection import learning_curve
from sklearn.pipeline import Pipeline

logger = logging.getLogger(__name__)


@dataclass
class LearningCurveConfig:
    """Configuration for learning curve diagnosis.

    Attributes:
        train_sizes: Fractions or absolute counts of training sizes to evaluate.
        cv: Number of cross-validation folds.
        scoring: Sklearn-compatible scoring metric.
        n_jobs: Parallel jobs for cross-validation. -1 uses all cores.
        shuffle: Whether to shuffle data before splitting (disable for time series).
        random_state: Seed for reproducibility.
    """

    train_sizes: np.ndarray = field(
        default_factory=lambda: np.linspace(0.1, 1.0, 10)
    )
    cv: int = 5
    scoring: str = "neg_mean_squared_error"
    n_jobs: int = -1
    shuffle: bool = True
    random_state: int = 42


class LearningCurveDiagnostic:
    """Generates and interprets learning curves to diagnose bias/variance.

    This diagnostic tool helps identify whether a model suffers from
    underfitting (high bias) or overfitting (high variance) by analyzing
    the gap and convergence pattern of training vs. validation curves.

    Args:
        estimator: Fitted or unfitted sklearn-compatible estimator or Pipeline.
        config: Configuration for the learning curve computation.

    Raises:
        ValueError: If estimator does not implement fit/predict interface.

    Example:
        >>> from sklearn.tree import DecisionTreeRegressor
        >>> diag = LearningCurveDiagnostic(DecisionTreeRegressor(max_depth=None))
        >>> diag.compute(X_train, y_train)
        >>> diag.plot(title="Decision Tree - Diagnóstico")
        >>> diagnosis = diag.diagnose()
        >>> print(diagnosis)
    """

    def __init__(
        self,
        estimator: BaseEstimator | Pipeline,
        config: LearningCurveConfig | None = None,
    ) -> None:
        if not (hasattr(estimator, "fit") and hasattr(estimator, "predict")):
            raise ValueError(
                "El estimador debe implementar fit() y predict(). "
                f"Recibido: {type(estimator).__name__}"
            )
        self.estimator = estimator
        self.config = config or LearningCurveConfig()
        self._train_sizes_abs: np.ndarray | None = None
        self._train_scores: np.ndarray | None = None
        self._val_scores: np.ndarray | None = None

    def compute(self, X: np.ndarray, y: np.ndarray) -> "LearningCurveDiagnostic":
        """Compute learning curve scores across training sizes.

        Args:
            X: Feature matrix of shape (n_samples, n_features).
            y: Target vector of shape (n_samples,).

        Returns:
            Self, for method chaining.

        Raises:
            ValueError: If X and y have incompatible shapes.
        """
        if X.shape[0] != y.shape[0]:
            raise ValueError(
                f"X tiene {X.shape[0]} muestras pero y tiene {y.shape[0]}. "
                "Deben coincidir."
            )

        logger.info(
            "Computando curvas de aprendizaje con cv=%d y scoring='%s'.",
            self.config.cv,
            self.config.scoring,
        )

        self._train_sizes_abs, self._train_scores, self._val_scores = learning_curve(
            self.estimator,
            X,
            y,
            train_sizes=self.config.train_sizes,
            cv=self.config.cv,
            scoring=self.config.scoring,
            n_jobs=self.config.n_jobs,
            shuffle=self.config.shuffle,
            random_state=self.config.random_state,
        )
        return self

    def _mean_std(
        self, scores: np.ndarray
    ) -> tuple[np.ndarray, np.ndarray]:
        """Convert negative scores to positive and compute mean/std."""
        positive = -scores if self.config.scoring.startswith("neg_") else scores
        return positive.mean(axis=1), positive.std(axis=1)

    def plot(
        self,
        title: str = "Curvas de Aprendizaje",
        figsize: tuple[int, int] = (10, 6),
        save_path: str | None = None,
    ) -> plt.Figure:
        """Plot training and validation learning curves with confidence bands.

        Args:
            title: Plot title.
            figsize: Figure dimensions (width, height) in inches.
            save_path: Optional path to save the figure.

        Returns:
            The matplotlib Figure object for further customization.

        Raises:
            RuntimeError: If compute() has not been called before plot().
        """
        if self._train_scores is None:
            raise RuntimeError(
                "Debés llamar compute() antes de plot(). "
                "Esa vaina no se genera sola."
            )

        train_mean, train_std = self._mean_std(self._train_scores)
        val_mean, val_std = self._mean_std(self._val_scores)

        fig, ax = plt.subplots(figsize=figsize)

        ax.plot(
            self._train_sizes_abs,
            train_mean,
            color="#2563EB",
            linewidth=2.5,
            marker="o",
            markersize=5,
            label="Error de Entrenamiento",
        )
        ax.fill_between(
            self._train_sizes_abs,
            train_mean - train_std,
            train_mean + train_std,
            alpha=0.15,
            color="#2563EB",
        )

        ax.plot(
            self._train_sizes_abs,
            val_mean,
            color="#DC2626",
            linewidth=2.5,
            marker="s",
            markersize=5,
            label="Error de Validación (CV)",
        )
        ax.fill_between(
            self._train_sizes_abs,
            val_mean - val_std,
            val_mean + val_std,
            alpha=0.15,
            color="#DC2626",
        )

        gap_final = val_mean[-1] - train_mean[-1]
        ax.annotate(
            f"Brecha final: {gap_final:.4f}",
            xy=(self._train_sizes_abs[-1], (val_mean[-1] + train_mean[-1]) / 2),
            xytext=(-120, 0),
            textcoords="offset points",
            fontsize=9,
            color="#374151",
            arrowprops={"arrowstyle": "->", "color": "#6B7280"},
        )

        ax.set_title(title, fontsize=14, fontweight="bold", pad=16)
        ax.set_xlabel("Tamaño del conjunto de entrenamiento", fontsize=11)
        ax.set_ylabel(
            self.config.scoring.replace("neg_", "").replace("_", " ").title(),
            fontsize=11,
        )
        ax.legend(fontsize=10, framealpha=0.9)
        ax.grid(True, linestyle="--", alpha=0.4)
        ax.yaxis.set_major_formatter(mticker.FormatStrFormatter("%.4f"))

        fig.tight_layout()

        if save_path:
            fig.savefig(save_path, dpi=150, bbox_inches="tight")
            logger.info("Figura guardada en '%s'.", save_path)
        else:
            plt.show()

        return fig

    def diagnose(self) -> dict[str, str | float]:
        """Diagnose model condition based on final curve values and gap.

        Returns:
            Dictionary with diagnosis keys:
                - 'condition': 'underfitting' | 'overfitting' | 'good_fit' | 'unknown'
                - 'train_error_final': Final training error.
                - 'val_error_final': Final validation error.
                - 'gap': Difference between val and train error.
                - 'recommendation': Human-readable action to take.

        Raises:
            RuntimeError: If compute() has not been called.
        """
        if self._train_scores is None:
            raise RuntimeError("Llamá compute() primero, parcero.")

        train_mean, _ = self._mean_std(self._train_scores)
        val_mean, _ = self._mean_std(self._val_scores)

        train_final = float(train_mean[-1])
        val_final = float(val_mean[-1])
        gap = val_final - train_final

        HIGH_ERROR_THRESHOLD = 0.10
        HIGH_GAP_THRESHOLD = 0.05

        if train_final > HIGH_ERROR_THRESHOLD and val_final > HIGH_ERROR_THRESHOLD:
            condition: Literal["underfitting", "overfitting", "good_fit", "unknown"] = (
                "underfitting"
            )
            recommendation = (
                "Bias alto. Aumentá la capacidad del modelo: más capas, "
                "mayor profundidad, features polinomiales o un modelo más complejo."
            )
        elif gap > HIGH_GAP_THRESHOLD:
            condition = "overfitting"
            recommendation = (
                "Varianza alta. Añadí regularización (L1/L2/Dropout), "
                "reducí complejidad del modelo, o conseguí más datos de entrenamiento."
            )
        elif train_final <= HIGH_ERROR_THRESHOLD and gap <= HIGH_GAP_THRESHOLD:
            condition = "good_fit"
            recommendation = (
                "El modelo generaliza bien. Revisá si el error irreducible "
                "justifica inversión adicional en features o datos."
            )
        else:
            condition = "unknown"
            recommendation = (
                "Patrón mixto. Analizá subgrupos de datos y revisá "
                "la distribución de train vs. producción."
            )

        logger.info(
            "Diagnóstico: %s | Train final: %.4f | Val final: %.4f | Gap: %.4f",
            condition,
            train_final,
            val_final,
            gap,
        )

        return {
            "condition": condition,
            "train_error_final": train_final,
            "val_error_final": val_final,
            "gap": gap,
            "recommendation": recommendation,
        }

🚨 Errores Comunes (y Cómo No Cagarla)

Error 1: Evaluar en datos de entrenamiento y llamarle "curva de aprendizaje"

El problema:

from sklearn.tree import DecisionTreeRegressor

model = DecisionTreeRegressor(max_depth=None)
model.fit(X_train, y_train)

# ❌ Esto no es diagnóstico, esto es trampa
train_errors = []
for n in range(10, len(X_train), 50):
    model.fit(X_train[:n], y_train[:n])
    pred = model.predict(X_train[:n])  # ❌ prediciendo sobre los mismos datos de fit
    train_errors.append(mean_squared_error(y_train[:n], pred))

plt.plot(train_errors)  # ❌ siempre bajará — no te dice nada

Por qué falla:

Un DecisionTreeRegressor sin max_depth memoriza perfectamente los datos de entrenamiento. El error en train siempre será cero o cercano a cero independientemente de n. Nunca verás el overfitting porque no estás midiendo generalización.

$$\mathcal{E}_{train}^{(\text{memorizado})} = \frac{1}{n}\sum_{i=1}^{n}\mathcal{L}(f_\theta(x_i), y_i) \approx 0 \quad \text{cuando } \text{depth} = \infty$$

import numpy as np
from sklearn.tree import DecisionTreeRegressor
from sklearn.metrics import mean_squared_error

np.random.seed(42)
X = np.random.randn(200, 5)
y = X[:, 0] ** 2 + np.random.randn(200) * 0.5

model = DecisionTreeRegressor(max_depth=None)
model.fit(X, y)

mse_train = mean_squared_error(y, model.predict(X))
print(f"MSE en train (mismo conjunto): {mse_train:.8f}")  # → 0.00000000
# No te enterás del overfitting hasta producción. De una catástrofe.

La solución:

from sklearn.model_selection import learning_curve
import numpy as np

train_sizes, train_scores, val_scores = learning_curve(
    estimator=DecisionTreeRegressor(max_depth=None),
    X=X,
    y=y,
    train_sizes=np.linspace(0.1, 1.0, 10),
    cv=5,                    # ✓ validación cruzada real
    scoring="neg_mean_squared_error",
    n_jobs=-1,
)
# ✓ val_scores mostrará la brecha real del overfitting

Error 2: Ignorar el efecto de cv insuficiente en datasets pequeños

El problema:

from sklearn.model_selection import learning_curve

# ❌ cv=2 con datasets pequeños produce estimaciones con alta varianza
train_sizes, train_scores, val_scores = learning_curve(
    estimator=model,
    X=X_small,    # ❌ 80 muestras con cv=2 → 40 para validar por fold
    y=y_small,
    cv=2,         # ❌ poquísimo, las curvas serán ruidosas
    scoring="neg_mean_squared_error",
)

Por qué falla:

Con cv=2 y $n=80$ muestras, cada fold de validación tiene solo 40 ejemplos. La estimación del error de validación tiene varianza altísima. Las curvas serán ruidosas y no podrás distinguir la señal real del ruido de muestreo.

$$\text{Var}\left[\hat{\mathcal{E}}_{val}\right] \propto \frac{1}{k \cdot n_{val}} \quad \text{donde } n_{val} = \frac{n}{k}$$

Con $k=2, n=80 \Rightarrow n_{val}=40$. Con $k=10, n=80 \Rightarrow n_{val}=8$. La solución es usar cv=5 como mínimo para datos pequeños y RepeatedKFold si el dataset es muy pequeño.

La solución:

from sklearn.model_selection import RepeatedKFold, learning_curve
import numpy as np

# ✓ RepeatedKFold reduce la varianza de la estimación en datasets pequeños
cv_strategy = RepeatedKFold(n_splits=5, n_repeats=3, random_state=42)

train_sizes, train_scores, val_scores = learning_curve(
    estimator=model,
    X=X_small,
    y=y_small,
    train_sizes=np.linspace(0.2, 1.0, 8),  # ✓ empieza en 0.2, no 0.05
    cv=cv_strategy,                          # ✓ 5 folds × 3 repeticiones
    scoring="neg_mean_squared_error",
    n_jobs=-1,
)

Error 3: Usar curvas de aprendizaje con datos temporales sin controlar el leak

El problema:

from sklearn.model_selection import learning_curve
from sklearn.linear_model import Ridge

# ❌ datos temporales con shuffle por defecto en learning_curve
train_sizes, train_scores, val_scores = learning_curve(
    estimator=Ridge(),
    X=X_temporal,  # ❌ datos ordenados en el tiempo
    y=y_temporal,
    cv=5,           # ❌ cv=5 estándar mezcla futuro y pasado
    scoring="neg_mean_squared_error",
)
# Las métricas serán optimistas → el modelo "vio el futuro" durante training

Por qué falla:

El KFold estándar con shuffle=True mezcla instancias de diferentes períodos. El modelo aprende con datos del futuro para predecir el pasado, generando data leakage temporal. En producción, cuando los datos llegan en orden cronológico, el modelo falla sistemáticamente.

$$P(\text{leak}) = 1 - \prod_{k=1}^{K} P(\text{val}_k > \max(\text{train}_k)) \approx 1 \quad \text{con shuffle}$$

La solución:

from sklearn.model_selection import TimeSeriesSplit, learning_curve
from sklearn.linear_model import Ridge
import numpy as np

# ✓ TimeSeriesSplit garantiza que validación siempre es "futuro" respecto al train
ts_cv = TimeSeriesSplit(n_splits=5)

train_sizes, train_scores, val_scores = learning_curve(
    estimator=Ridge(),
    X=X_temporal,
    y=y_temporal,
    train_sizes=np.linspace(0.2, 1.0, 8),
    cv=ts_cv,       # ✓ respeta el orden temporal
    scoring="neg_mean_squared_error",
    n_jobs=-1,
)

Error 4: No usar Pipeline (leakage en preprocessing dentro del CV)

El problema:

from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import Ridge
from sklearn.model_selection import learning_curve

# ❌ scaler fitteado sobre TODO X antes del split
scaler = StandardScaler()
X_scaled = scaler.fit_transform(X)  # ❌ información del test contamina el scaler

train_sizes, train_scores, val_scores = learning_curve(
    estimator=Ridge(),
    X=X_scaled,     # ❌ ya hay leakage aquí
    y=y,
    cv=5,
)

Por qué falla:

Al hacer fit_transform sobre todo X antes de la curva, la media y desviación estándar del scaler incluyen información del conjunto de validación. El modelo "conoce" estadísticas globales que no debería ver. Eso infla métricas de validación.

La solución:

from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import Ridge
from sklearn.pipeline import Pipeline
from sklearn.model_selection import learning_curve
import numpy as np

# ✓ Pipeline garantiza que el scaler solo se fittea sobre train en cada fold
pipeline = Pipeline([
    ("scaler", StandardScaler()),  # ✓ fit solo en train de cada fold
    ("model", Ridge(alpha=1.0)),
])

train_sizes, train_scores, val_scores = learning_curve(
    estimator=pipeline,
    X=X,      # ✓ sin escalar — Pipeline lo maneja internamente
    y=y,
    train_sizes=np.linspace(0.1, 1.0, 10),
    cv=5,
    scoring="neg_mean_squared_error",
    n_jobs=-1,
)

🎯 Resumen Ejecutivo

Over/Underfitting no es "ese concepto de la primera clase". Es el diagnóstico de la salud de tu modelo. El problema central es minimizar:

$$\mathbb{E}\left[\mathcal{L}(f_\theta(x), y)\right] = \text{Bias}^2[f_\theta(x)] + \text{Var}[f_\theta(x)] + \sigma^2_\epsilon$$

Y la herramienta que te permite ver dónde está el problema es la curva de aprendizaje:

$$\mathcal{E}(n) = \frac{1}{n} \sum_{i=1}^{n} \mathcal{L}(f_\theta(x_i), y_i) \quad \text{evaluada en train y validación}$$

Con todas las implicaciones que eso conlleva:

  • Bias alto (underfitting): Las dos curvas convergen hacia un error alto. Más datos no ayudan. Hay que aumentar la capacidad del modelo o hacer feature engineering.
  • Varianza alta (overfitting): Gran brecha entre train y validación. Más datos sí ayudan. Regularización, dropout o simplificación del modelo son el camino.
  • Ruido irreducible: Define el piso. Si tu error de validación no baja de $\sigma^2_\epsilon$, ningún modelo lo va a resolver — revisá la calidad de tus datos.

De una, hágale. Con esta base de bias-varianza + curvas de aprendizaje production-ready, tenés el lenguaje para diagnosticar cualquier modelo antes de mandarlo a producción. No más "reentrenar y rezar".
