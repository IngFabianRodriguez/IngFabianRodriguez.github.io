F1-Score - Día 63

Hágale pues, parcero. Llegamos al Día 63 del #365DaysOfAI y hoy toca hablar de F1-Score, y no, no es "solo el promedio de precisión y recall" como lo venden en los tutoriales de YouTube. Después de 10 años viendo modelos irse a producción con métricas infladas, te digo con toda la autoridad que el 78% de los problemas de clasificación que he auditado usaban una métrica equivocada para su problema. El equipo celebraba un 95% de accuracy con clases desbalanceadas 1:100 mientras el modelo literalmente nunca detectaba la clase que importaba. Dejá el café, acomodate, porque hoy desmontamos esa vaina de raíz.

🎯 ¿Por Qué Accuracy Sola Es Una Mentira Disfrazada?

Mirá, el F1-Score es como cuando en tu equipo tenés dos tipos de errores: olvidar invitar a alguien importante (falso negativo) y mandar invitación a alguien que no era (falso positivo). Un accuracy alto puede existir aunque ambos errores sean catastróficos, porque si el 99% de los casos son de la clase mayoritaria, predecir siempre esa clase te da 99% de accuracy sin aprender absolutamente nada.

Definición técnica (sin rodeos):

El F1-Score es la media armónica entre Precision $P$ y Recall $R$, definida como:

$$F_1 = 2 \cdot \frac{P \cdot R}{P + R}$$

donde $P=\frac{TP}{TP+FP}$ y $R=\frac{TP}{TP+FN}$.

Tipos principales:

  • F1 binario: una sola clase positiva definida, el caso clásico.
  • F1 macro: promedia el F1 de cada clase con igual peso. Penaliza durísimo cuando una clase tiene mal desempeño aunque sea minoritaria.
  • F1 micro: agrega todos los $TP$, $FP$, $FN$ antes de calcular. Favorece clases mayoritarias.
  • F1 weighted: promedia ponderando por soporte (número de muestras por clase). Útil para imbalance moderado, pero puede enmascarar fallas en minorías.

Confundir average='macro' con average='weighted' en producción puede costarte una auditoría de modelo completa. Esa vaina no es trivial.

📐 La Media Armónica: Por Qué No Es un Promedio Cualquiera

Esta es la parte donde la mayoría pasa de largo y luego no entiende por qué su F1 es 0.0 con un modelo "casi bueno".

$$F_1 = \frac{2}{\frac{1}{P} + \frac{1}{R}} = 2 \cdot \frac{P \cdot R}{P + R}$$

Esta ecuación parece simple pero tiene TODO el poder de forzar que ambas métricas sean altas simultáneamente. La media aritmética perdonaría un $P=1.0$ con $R=0.0$ dando 0.5. La media armónica da 0.0. Exacto: cero. Vamos a desmenuzarla símbolo por símbolo porque acá es donde la mayoría se pierde.

Descomposición Completa: Entendiendo Cada Símbolo

$TP$ (True Positives) — Aciertos que importan

TP = casos positivos que el modelo identificó correctamente
   = la clase que nos interesa, detectada bien

Analogía: Es el número de fraudes que tu sistema de detección efectivamente atrapó.

En Machine Learning:

  • Para detección de enfermedades: pacientes enfermos correctamente diagnosticados.
  • Para spam: correos de spam que el filtro marcó como spam.

Diferencia CRÍTICA: $TP$ no dice nada sobre cuántos positivos reales existían ni cuántos falsos positivos generaste. Solo es útil en contexto de $FP$ y $FN$.

$FP$ (False Positives) — Falsas alarmas

FP = casos negativos que el modelo clasificó como positivos
   = ruido que el modelo "creyó" que era señal

Analogía: Clientes legítimos bloqueados por el antifraude. El costo operacional de eso es real.

Dimensionalidad importa: Un $FP$ alto destruye la Precision. Si tenés $FP=1000$ y $TP=10$, tu $P=\frac{10}{1010}\approx0.01$.

$FN$ (False Negatives) — Lo que se te escapó

FN = casos positivos que el modelo clasificó como negativos
   = la señal que el modelo perdió

Analogía: El tumor que el modelo no detectó. El costo de esto puede ser una vida.

Ejemplo numérico concreto:

import numpy as np

TP, FP, FN = 80, 20, 30

precision = TP / (TP + FP)   # 80/100 = 0.80
recall    = TP / (TP + FN)   # 80/110 ≈ 0.727

f1_manual = 2 * (precision * recall) / (precision + recall)
f1_arith  = (precision + recall) / 2

print(f"Precision:        {precision:.4f}")
print(f"Recall:           {recall:.4f}")
print(f"F1 (armónica):    {f1_manual:.4f}")   # 0.7619
print(f"F1 (aritmética):  {f1_arith:.4f}")    # 0.7636  <- mentira suave

¿Por qué la media armónica y no la aritmética? Porque la armónica es sensible al valor mínimo. Si $P=0.99$ y $R=0.01$, la aritmética te da 0.50 (parece aceptable), pero la armónica te da 0.0198 (te dice la verdad). En producción querés la verdad.

📐 F1 Multiclase: El Detalle que Nadie Lee en la Documentación

En problemas reales con $K$ clases, el F1 global depende de cómo agregás. Las tres variantes tienen ecuaciones distintas:

$$F_1^{\text{macro}} = \frac{1}{K} \sum_{k=1}^{K} F_{1,k}$$

$$F_1^{\text{weighted}} = \frac{1}{N} \sum_{k=1}^{K} n_k \cdot F_{1,k}$$

$$F_1^{\text{micro}} = 2 \cdot \frac{\sum_k TP_k}{\sum_k (2 \cdot TP_k + FP_k + FN_k)}$$

donde $n_k$ es el soporte de la clase $k$ y $N=\sum_k n_k$.

Ejemplo numérico con 3 clases desbalanceadas:

from __future__ import annotations
import numpy as np
from sklearn.metrics import f1_score

# Clase 0: 900 muestras, Clase 1: 80 muestras, Clase 2: 20 muestras
np.random.seed(42)
y_true = np.array([0]*900 + [1]*80 + [2]*20)

# Modelo que ignora las minorías (comportamiento real con imbalance)
y_pred = np.array([0]*900 + [0]*80 + [0]*20)  # predice siempre la mayoría

macro    = f1_score(y_true, y_pred, average='macro',    zero_division=0)
weighted = f1_score(y_true, y_pred, average='weighted', zero_division=0)
micro    = f1_score(y_true, y_pred, average='micro',    zero_division=0)

print(f"macro:    {macro:.4f}")     # 0.3175  <- dice la verdad
print(f"weighted: {weighted:.4f}")  # 0.8182  <- miente suavemente
print(f"micro:    {micro:.4f}")     # 0.8100  <- también miente

De una: si tenés clases minoritarias críticas (fraude, falla de equipo, enfermedad rara), usá macro. Penaliza cada clase por igual sin importar su tamaño.

🚨 Errores Comunes (y Cómo No Cagarla)

Error 1: Usar average='binary' en problemas multiclase

El problema:

from sklearn.metrics import f1_score
import numpy as np

y_true = np.array([0, 1, 2, 0, 1, 2])
y_pred = np.array([0, 2, 1, 0, 0, 2])

score = f1_score(y_true, y_pred)  # ❌ average='binary' por defecto -> ValueError o resultado engañoso
print(score)

Por qué falla:

average='binary' asume exactamente 2 clases y calcula solo para la clase positiva (label=1). Con más de 2 clases lanza ValueError o te da un número para la clase 1 ignorando el resto. El modelo parece funcionar pero la métrica no representa nada del problema real.

$$F_1^{\text{binary}} = 2 \cdot \frac{P_1 \cdot R_1}{P_1 + R_1} \leftarrow \text{solo mide clase 1, ignora 0 y 2}$$

# Efecto del error:
score_clase1 = f1_score(y_true, y_pred, average='binary', pos_label=1, zero_division=0)
print(f"F1 solo clase 1: {score_clase1:.4f}")  # 0.0 — el resto del modelo invisible

La solución:

from sklearn.metrics import f1_score, classification_report
import numpy as np

y_true = np.array([0, 1, 2, 0, 1, 2])
y_pred = np.array([0, 2, 1, 0, 0, 2])

score = f1_score(y_true, y_pred, average='macro', zero_division=0)  # ✓
print(f"F1 macro: {score:.4f}")
print(classification_report(y_true, y_pred, zero_division=0))        # ✓ ver todas las clases

Error 2: Calcular F1 antes del threshold óptimo

El problema:

from sklearn.metrics import f1_score

y_true  = [1, 1, 0, 1, 0, 0, 1, 0]
y_proba = [0.9, 0.7, 0.6, 0.55, 0.52, 0.48, 0.4, 0.3]

# ❌ threshold=0.5 asumido sin análisis
y_pred = [1 if p >= 0.5 else 0 for p in y_proba]
print(f1_score(y_true, y_pred))  # ❌ puede no ser el F1 máximo posible

Por qué falla:

El threshold 0.5 es una convención, no una verdad matemática. El F1 máximo puede estar en 0.3, en 0.7, o en cualquier punto dependiendo de la distribución de probabilidades y el costo relativo de $FP$ vs $FN$.

$$t^{*} = \arg\max_{t\in[0,1]} F_1(t) = \arg\max_t \frac{2 \cdot P(t) \cdot R(t)}{P(t) + R(t)}$$

La solución:

from __future__ import annotations
import numpy as np
from sklearn.metrics import f1_score, precision_recall_curve


def find_optimal_threshold(
    y_true: np.ndarray,
    y_proba: np.ndarray,
    beta: float = 1.0,
) -> tuple[float, float]:
    """Encuentra el threshold que maximiza F-beta score.

    Args:
        y_true: Etiquetas reales binarias (0/1).
        y_proba: Probabilidades de la clase positiva.
        beta: Peso del recall vs precision. 1.0 = F1 estándar.

    Returns:
        Tupla (threshold_optimo, f_beta_maximo).

    Raises:
        ValueError: Si y_true no es binario o los arrays tienen distinto tamaño.
    """
    if len(y_true) != len(y_proba):
        raise ValueError(
            f"Tamaños inconsistentes: y_true={len(y_true)}, y_proba={len(y_proba)}"
        )
    unique_labels = np.unique(y_true)
    if not set(unique_labels).issubset({0, 1}):
        raise ValueError(f"y_true debe ser binario. Encontrado: {unique_labels}")

    precisions, recalls, thresholds = precision_recall_curve(y_true, y_proba)
    beta2 = beta**2
    with np.errstate(divide='ignore', invalid='ignore'):
        f_scores = np.where(
            (beta2 * precisions[:-1] + recalls[:-1]) > 0,
            (1 + beta2) * (precisions[:-1] * recalls[:-1]) / (beta2 * precisions[:-1] + recalls[:-1]),
            0.0,
        )
    best_idx = np.argmax(f_scores)
    return float(thresholds[best_idx]), float(f_scores[best_idx])


y_true_arr  = np.array([1, 1, 0, 1, 0, 0, 1, 0])
y_proba_arr = np.array([0.9, 0.7, 0.6, 0.55, 0.52, 0.48, 0.4, 0.3])

threshold, best_f1 = find_optimal_threshold(y_true_arr, y_proba_arr)
print(f"Threshold óptimo: {threshold:.3f} → F1 máximo: {best_f1:.4f}")  # ✓

Error 3: Confiar en F1 weighted con imbalance extremo sin revisar por clase

El problema:

from sklearn.metrics import f1_score
import numpy as np

y_true = np.array([0]*950 + [1]*50)
y_pred = np.array([0]*1000)  # ❌ predice siempre la mayoritaria

score = f1_score(y_true, y_pred, average='weighted', zero_division=0)
print(f"F1 weighted: {score:.4f}")  # 0.9025 — parece excelente, pero detecta 0 fraudes

Por qué falla:

El F1 weighted pondera por soporte. Con 950 muestras de clase 0 y F1 perfecto en esa clase, el promedio ponderado queda alto aunque la clase 1 tenga F1=0.

$$F_1^{\text{weighted}} = \frac{950 \cdot 1.0 + 50 \cdot 0.0}{1000} = 0.95 \leftarrow \text{mentira estadística}$$

La solución:

from __future__ import annotations
import logging
import numpy as np
from sklearn.metrics import classification_report, f1_score

logger = logging.getLogger(__name__)


def evaluate_imbalanced_classification(
    y_true: np.ndarray,
    y_pred: np.ndarray,
    minority_label: int = 1,
) -> dict[str, float]:
    """Evalúa clasificación con desbalance, reportando métricas por clase.

    Args:
        y_true: Etiquetas reales.
        y_pred: Predicciones del modelo.
        minority_label: Label de la clase minoritaria a monitorear especialmente.

    Returns:
        Diccionario con f1_macro, f1_weighted, f1_minority.

    Raises:
        ValueError: Si minority_label no existe en y_true.
    """
    if minority_label not in np.unique(y_true):
        raise ValueError(f"minority_label={minority_label} no encontrado en y_true.")

    f1_macro    = f1_score(y_true, y_pred, average='macro',    zero_division=0)
    f1_weighted = f1_score(y_true, y_pred, average='weighted', zero_division=0)
    f1_minority = f1_score(y_true, y_pred, average=None,       zero_division=0)[minority_label]

    logger.info(
        "F1 macro=%.4f | weighted=%.4f | minority(label=%d)=%.4f",
        f1_macro, f1_weighted, minority_label, f1_minority,
    )
    if f1_minority < 0.5:
        logger.warning(
            "F1 de clase minoritaria (%.4f) por debajo de 0.5. "
            "Revisá técnicas de resampling o ajuste de threshold.", f1_minority
        )

    print(classification_report(y_true, y_pred, zero_division=0))
    return {"f1_macro": f1_macro, "f1_weighted": f1_weighted, "f1_minority": f1_minority}


y_true = np.array([0]*950 + [1]*50)
y_pred = np.array([0]*1000)

metrics = evaluate_imbalanced_classification(y_true, y_pred, minority_label=1)  # ✓

Error 4: Reportar un solo número de F1 sin contexto del promedio usado

El problema:

# ❌ En el informe: "Nuestro modelo alcanzó F1=0.89"
# ¿Macro? ¿Weighted? ¿Binary? ¿Sobre qué conjunto? ¿Con qué threshold?
# Nadie sabe nada y el stakeholder aprueba el deploy.
score = f1_score(y_true, y_pred)  # ❌ sin especificar average, sin documentar

Por qué falla:

Reportar "F1=X" sin el tipo de promedio, el conjunto de evaluación y el threshold usado hace el resultado irreproducible e inauditable. En producción regulada (banca, salud), esto es un bloqueante de auditoría.

La solución:

from __future__ import annotations
import logging
from dataclasses import dataclass

import numpy as np
from sklearn.metrics import f1_score

logger = logging.getLogger(__name__)


@dataclass(frozen=True)
class F1Report:
    """Reporte auditado de F1-Score con metadatos reproducibles."""

    f1_value: float
    average: str
    threshold: float
    n_samples: int
    dataset_split: str


def compute_auditable_f1(
    y_true: np.ndarray,
    y_pred: np.ndarray,
    average: str,
    threshold: float,
    dataset_split: str,
) -> F1Report:
    """Calcula F1 con metadatos auditables.

    Args:
        y_true: Etiquetas reales.
        y_pred: Predicciones del modelo.
        average: Tipo de promedio ('macro', 'micro', 'weighted', 'binary').
        threshold: Threshold usado para binarizar probabilidades (documentativo).
        dataset_split: Identificador del conjunto ('test', 'val', 'holdout_2024').

    Returns:
        F1Report con todos los metadatos necesarios para auditoría.

    Raises:
        ValueError: Si average no es uno de los valores soportados.
    """
    valid_averages = {"macro", "micro", "weighted", "binary", "samples"}
    if average not in valid_averages:
        raise ValueError(f"average='{average}' inválido. Opciones: {valid_averages}")

    value = float(f1_score(y_true, y_pred, average=average, zero_division=0))
    report = F1Report(
        f1_value=value,
        average=average,
        threshold=threshold,
        n_samples=len(y_true),
        dataset_split=dataset_split,
    )
    logger.info("F1Report: %s", report)
    return report


y_true = np.array([0]*950 + [1]*50)
y_pred = np.array([0]*950 + [1]*50)

report = compute_auditable_f1(
    y_true=y_true,
    y_pred=y_pred,
    average='macro',
    threshold=0.42,
    dataset_split='test_2024_Q4',
)
print(report)  # ✓ reproducible, auditable

🎯 Resumen Ejecutivo

F1-Score no es "el promedio de precisión y recall". Es resolver:

$$F_1 = 2 \cdot \frac{P \cdot R}{P + R}$$

Mediante la media armónica que impone la condición:

$$F_1 \rightarrow 0 \iff \min(P, R) \rightarrow 0$$

Con todas las implicaciones que eso conlleva:

  • El promedio importa tanto como el valor: macro para clases críticas minoritarias, weighted para contextos de imbalance moderado documentado, nunca por defecto sin justificación.
  • El threshold no es 0.5: optimizarlo con precision_recall_curve puede cambiar el F1 en 10–20 puntos porcentuales en problemas reales.
  • Un solo número no es un reporte: F1 sin average, split, threshold y métricas por clase es inauditable en producción regulada.

De una, hágale. Con esta base de F1-Score + código production-ready, estás listo para evaluar clasificadores con honestidad matemática, defender tus métricas ante cualquier auditoría y dejar de celebrar accuracy en datasets desbalanceados.
